title: 常用英文词（非专有名词）
---

下面这些不是领域专有名词，而是一些常见英文词。它们大多有对应中文，但直译过来要么别扭、要么不够精准，所以正文里也直接用了英文。

- **review** — 审阅 / 评审。指对方案的每条细项、每个 commit 逐条过一遍，确认范围、边界和风险后再放行。中文「审核」偏向行政审批，和这里「逐条讨论、发现问题就修正」的含义不完全贴合，故保留英文。
- **brainstorm** — 头脑风暴，发散地把模糊想法聊清楚。既是通用动词，也是我们用的 `ce-brainstorm` 命令；「头脑风暴」略显冗长，正文多直接用 brainstorm。
- **plan** — 把需求整理成可执行、可评审的方案。这里同时对应 `ce-plan` 命令，作动词时（「先 brainstorm、再 plan」）用英文更连贯。
- **helper** — 小的辅助函数 / 工具函数。中文「助手」「辅助函数」都不够自然，正文用 helper 指这类不含业务逻辑的小函数（如运行时路径 helper）。
- **wrapper** — 薄封装层。在底层调用（如 Bun）外面包一层统一入口；中文「包装器」偏生硬，故用 wrapper。
- **normalize** — 归一 / 规范化。把不同来源的值统一成同一种形式（如路径 normalize，或把不同模型的差异 normalize 成统一表示）。作动词时英文更顺口，故保留。
- **dummy** — 占位用的、假的。如 dummy React 项目：只为提供一个「看起来像真的」样例，本身不含真实逻辑。中文「假的」不够精准，「占位」React 项目不好听，故用 dummy。
- **fixture** — 测试夹具，即测试用的固定数据或样例工程。中文「夹具」不直观，英文在测试语境里更通用，故保留。
- **smoke test** — 冒烟测试，即最基础的一条「能跑起来、主流程不崩」的测试。英文说法更常见，正文保留。
- **checkout** — （Git）检出，把代码拉到本地。如「新 checkout 的仓库」指刚克隆下来的干净仓库。属 Git 惯用语，用英文更自然。
- **contributors** — 贡献者 / 参与项目的开发者。英文在开源语境里更常用，正文多保留。
- **commit-sized** — 「一个 commit 大小的」。形容刚好适合放进单次提交的小单元（如一个 U 单元）。中文很难简洁表达，故用 commit-sized。
- **item** — 一条 / 一项。正文里指列表中的单条记录，如转录区里的一条消息（对应代码 `BodyEntry`）或方案文档里可勾选的一条内容；中文「条目」偏书面，故直接用 item。
