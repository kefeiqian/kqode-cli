`v0.1.1` 的主题不是扩大功能面，而是把 KQode 早期研发中几个会反复出现的“边界”先立住：TUI session 如何收尾，构建环境如何表达，版本号如何发布，Agent 作者工作流如何避免漂移。

![v0.1.1 决策与延期事项](../images/v0-1-1-summary/decisions-and-deferrals.png)

## 最终发布落点

发布 commit [`24ae0934`](https://github.com/kefeiqian/KQode/commit/24ae0934b368e36beb8078c99f53c70c3e71abca) 把 manifest 更新到 `0.1.1`。根目录 [`Cargo.toml`](https://github.com/kefeiqian/KQode/blob/24ae0934b368e36beb8078c99f53c70c3e71abca/Cargo.toml) 中的版本是：

```toml
[package]
name = "KQode"
version = "0.1.1"
edition = "2024"
```

TUI package [`tui/package.json`](https://github.com/kefeiqian/KQode/blob/24ae0934b368e36beb8078c99f53c70c3e71abca/tui/package.json) 也同步到同一版本：

```json
{
  "name": "@kqode/tui",
  "version": "0.1.1",
  "private": true,
  "type": "module",
```

npm launcher [`packaging/npm/kqode/package.json`](https://github.com/kefeiqian/KQode/blob/24ae0934b368e36beb8078c99f53c70c3e71abca/packaging/npm/kqode/package.json) 同样是 `0.1.1`：

```json
{
  "name": "@kqode/kqode-cli",
  "version": "0.1.1",
  "description": "KQode — a Rust-core coding-agent harness with a TypeScript Ink terminal UI.",
```

这验证了上一篇的设计：版本发布不应该靠手工逐个改文件，而应该通过 `set-version` 的锁步语义完成。

## 决策一：退出总结卡片先只展示可信数据

**决策**：`v0.1.1` 的 exit summary card 只展示 `Duration` 和 `Changes`，`Cost`、`Tokens`、`Resume` 暂时省略。

**为什么**：summary card 是 session 结束时用户最容易信任的信息。如果一开始就放入没有真实数据源的占位内容，后续很难建立“卡片上每一行都可靠”的产品契约。先只展示 duration 和 Git line delta，可以验证 terminal teardown、TTY 检测、宽度退化和 baseline 计算，而不伪造 cost / token。

**延期**：token、cost、resume command、checkpoint 和 trace evidence 需要等 provider、session store、pricing 和 replay 结构稳定后再接入。

## 决策二：环境语义统一，但实现保持生态原生

**决策**：TUI 使用 `__DEV__` / `__TEST__` / `__PROD__`，Rust core 使用 `build.rs` 暴露 `__DEV__` / `__PROD__`，并用内建 `cfg(test)` 表达 test。

**为什么**：TypeScript 打包器需要裸全局变量做 dead-code elimination；Rust 编译器需要 cfg 做条件编译。强行让两边共享同一种技术实现并不现实，但共享命名和语义很重要。这样后续看到 `prod`、`dev`、`test` 时，不需要猜这是 TUI 的概念还是 Rust 的概念。

**延期**：当前只是统一 build flags，没有引入更完整的 release profile、feature matrix 或 provider-specific build target。那些应该等 package / eval / provider 层更稳定后再加。

## 决策三：版本号只能通过 `xtask` 锁步修改

**决策**：新增 `cargo xtask set-version <MAJOR.MINOR.PATCH>`，集中更新 root Cargo、`xtask` Cargo、TUI package 和 npm launcher package。

**为什么**：KQode 横跨 Rust、TypeScript 和 npm 分发。如果版本号不一致，用户会看到 binary、package 和 tag 互相矛盾。把写操作收口到 `xtask` 后，release skill、IDE profile 和人工命令行都能走同一条路径。

**延期**：这个命令只处理 manifest 和 `Cargo.lock`。它没有负责创建 Git tag、push、GitHub Release 或 npm publish，因为这些是远端发布动作，需要明确的人类确认和 CI / workflow 配合。

## 决策四：Agent skill 记录流程，不复制业务逻辑

**决策**：`kqode-version-bump` skill 负责交互和提醒，但实际写版本号时调用 `cargo xtask set-version`。

**为什么**：Agent skill 很适合保存“先读当前版本、展示候选、询问用户、提醒 tag”这类流程知识，但不应该成为第二套业务逻辑。真正改 manifest 的代码必须只有一处，否则一旦新增 manifest，skill 和 `xtask` 很容易不一致。

**延期**：skill 不自动 tag、不 push、不发布。它可以提示下一步，但 release 边界仍然需要用户明确授权。

## 决策五：文档要固定到 commit，而不是移动的 `main`

**决策**：blog 新文章 skill 要求引用源代码时使用 pinned commit permalink。

**为什么**：开发日志描述的是某个版本的实现。如果链接到 `main`，代码未来变化后，读者点开链接看到的内容就不再是文章讨论的内容。`v0.1.1` 文章大量引用 `cc82123e`、`f13b6d1b`、`9af3d5f0` 和 `4aefb513` 的文件，必须固定到当时的树。

**延期**：本次只写中文 release 文章。英文 locale 可以后续通过 blog 翻译 workflow 补齐，但不应该阻塞中文开发日志落地。

## docs-only commit 的收获

[`d1813eca`](https://github.com/kefeiqian/KQode/commit/d1813eca6d8219573feeafad10403bb758f71072) 的 blog 重组，让教程从单层列表变成 U1 / U2 结构；[`d387c78e`](https://github.com/kefeiqian/KQode/commit/d387c78e0c00ced6218c5b0dadf05c371721220f) 的 LLM provider streaming chat plan，为后续 provider 层实现提前划边界；[`17b51456`](https://github.com/kefeiqian/KQode/commit/17b51456a479ab5de20af403bf1668788099d076) 则把 homepage plan、`README.md` 和 repo instructions 重新对齐。

这些 commit 不直接产生 runtime 功能，但它们降低了后续实现时的解释成本。KQode 现在仍处在项目基础阶段，文档和计划不是附属品，而是用来约束下一步实现的产品边界。

## 下一步

从 `v0.1.1` 往后，最自然的方向是把 summary card 后面的真实数据源补上：session log、provider token usage、cost accounting、resume command 和 trace evidence。同时，LLM provider streaming chat plan 已经准备好，下一阶段可以开始让 Rust core 和 TUI 在 streaming protocol 上真正闭环。
