U8 完成了第一个端到端 ACK 回路。用户在 Ink composer 输入 prompt，按 Enter 后进入前端 FIFO queue，经由 U7 的 backend client 发给 Rust ACK 后端，收到结果后追加到正文 transcript。实现 commit 是 [`bcc8130e137f37b8dc0884918768f3b1fc5fc1ef`](https://github.com/kefeiqian/KQode/commit/bcc8130e137f37b8dc0884918768f3b1fc5fc1ef)。

![U8 完成后的 ACK transcript](../../images/u8-submit-ack-wiring/u8-summary-transcript.png)

## 关键决策与为什么

1. 使用 FIFO queue，而不是并发发送。原因是 U8 先保证顺序和结果归属。
2. prompt 立即进入正文，而不是等 ACK。原因是用户需要即时反馈。
3. 所有正文文本先 `sanitizeDisplayText`。原因是 terminal control 字符不能被执行。
4. UI / state 层禁止引用 process 启动代码。原因是 source mode 和后续 packaged mode 必须共享 backend client seam。

## 有意延后

U8 没有做真实 LLM、assistant stream、工具调用、session persistence、retry 策略或 standalone executable。它只验证本地 ACK 往返；这些延后项分别留给后续 agent runtime、会话系统和 U9 打包路径。
