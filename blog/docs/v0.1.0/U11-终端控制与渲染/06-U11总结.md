title: 6. U11 总结
---

U11 的真实 diff 很聚焦：终端控制 helper、对应测试、产品版本解析。它没有把鼠标追踪或 bootstrap 接线一并做完，而是先把“写入终端控制序列”这件事变成可测试基础设施。

## 关键决策与为什么

- 使用独立 `tui/src/libs/terminal/` 模块承载 escape sequence。为什么：这些能力是 terminal side effect，不应该散落在 React component 里；集中封装后，后续 bootstrap 只需要处理生命周期顺序。
- 每个写入函数都做 `TTY` 守卫。为什么：pipe、测试和日志捕获不应该混入不可见控制序列；把 guard 放在 helper 内部，比要求调用者记住更安全。
- 对 OSC / DEC sequence 做 exact byte 测试。为什么：终端控制 bug 往往是 invisible byte 错误，TypeScript 类型系统不能发现少写 `BEL` 或写错 mode number。
- 用 OSC 111 恢复背景，而不是查询原背景。为什么：背景查询需要异步读取终端响应和 timeout 处理，超出 U11 的粒度；reset sequence 是更小、更稳的恢复策略。
- 新增 `resolveProductVersion` 支持 build-time 注入。为什么：packaged TUI 不应该依赖仓库根目录读取 Cargo manifest；但 source mode 仍然需要保留本地开发路径。

```ts
const injected = process.env.KQODE_VERSION;
if (injected !== undefined && injected.length > 0) {
  return injected;
}
```

## 明确推迟的事

U11 推迟了三件事。第一，鼠标追踪没有在这个提交出现；后续如果要支持点击、滚轮或 selection-aware UI，需要新增独立 helper 和退出恢复策略。第二，bootstrap 生命周期接线没有在这个 diff 中修改；终端 helper 已经准备好，但接入启动和退出路径应作为单独提交 review。第三，终端能力探测没有做；U11 选择“unsupported terminal 忽略 sequence”的渐进增强模式，而不是构建完整 capability detection。

## 这个单元留下的基础

最终，U11 让 KQode 的 TUI 从“能打印内容”向“尊重终端环境”前进了一步。后续所有全屏体验、主题背景、窗口标题和 release 版本展示，都可以复用这些 helper，而不是重新手写 escape sequence。
