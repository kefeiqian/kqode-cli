U11 新增 [`terminalBackground.ts`](https://github.com/kefeiqian/KQode/blob/97a9a65ddb34dce8404927ea0cd26f6860b2ed00/tui/src/libs/terminal/terminalBackground.ts)，用于设置和恢复终端默认背景色。这里使用两个 OSC 序列：OSC 11 设置背景，OSC 111 恢复终端默认背景。

```ts
const SET_BACKGROUND_PREFIX = '\u001B]11;';
const SET_BACKGROUND_SUFFIX = '\u0007';

/**
 * The OSC 111 escape sequence that resets the terminal default background color
 * to whatever the user's terminal emulator uses by default.
 */
export const RESET_BACKGROUND_SEQUENCE = '\u001B]111\u0007';

/** Builds the OSC 11 escape sequence that sets the terminal default background to `color`. */
export function buildSetBackgroundSequence(color: string): string {
  return `${SET_BACKGROUND_PREFIX}${color}${SET_BACKGROUND_SUFFIX}`;
}
```

## 为什么要改终端背景

Ink 渲染的是字符区域，但终端窗口里经常还有 padding、空白区域和 resize 后尚未覆盖的区域。如果只在 React component 里设置背景，UI 面板内部是 Dracula 深色，窗口边缘却仍然是用户默认背景，视觉上会像一个“贴在终端里的色块”。

OSC 11 处理的是 terminal default background。它让 KQode 会话期间整个终端背景都与主题一致，尤其是在全屏 TUI、备用屏幕和 resize 场景下更稳定。

## 为什么恢复用 OSC 111

一个看似更“完美”的方案是：启动时查询原背景，退出时恢复原值。但 U11 没有这么做，而是直接暴露 OSC 111 reset：

```ts
export function resetTerminalBackground(stream: NodeJS.WriteStream = process.stdout): void {
  if (!stream.isTTY) {
    return;
  }

  stream.write(RESET_BACKGROUND_SEQUENCE);
}
```

原因是背景查询不是一个简单同步 API。OSC 查询需要终端回包，程序要从 stdin 读取响应，还要处理 timeout、用户输入竞争、不同 terminal emulator 格式差异，以及非交互环境。对于 U11 这个阶段，查询原背景会把“设置背景”扩大成“终端能力探测子系统”，超出了提交粒度。

OSC 111 的取舍是：不保存原始值，但让 terminal emulator 自己恢复默认背景。它更简单，也更容易在所有退出路径里调用。

## 为什么测试 exact bytes

[`terminalBackground.test.ts`](https://github.com/kefeiqian/KQode/blob/97a9a65ddb34dce8404927ea0cd26f6860b2ed00/tui/src/libs/terminal/__tests__/terminalBackground.test.ts) 没有只测试“调用了 write”，而是断言具体序列：

```ts
expect(buildSetBackgroundSequence('#282A36')).toBe('\u001B]11;#282A36\u0007');
expect(RESET_BACKGROUND_SEQUENCE).toBe('\u001B]111\u0007');
```

这能防止两类回归：一是 OSC 编号写错，例如把 `11` 和 `111` 混淆；二是结束符丢失，导致终端把后续输出继续吞进 OSC payload。终端控制代码最危险的 bug 往往不是 TypeScript 类型错误，而是 invisible byte 错误，所以这里需要非常具体的断言。
