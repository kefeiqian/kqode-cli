U11 对应 commit [`97a9a65d`](https://github.com/kefeiqian/KQode/commit/97a9a65ddb34dce8404927ea0cd26f6860b2ed00)，主题是 `feat(tui): integrate terminal control and rendering robustness`。我按 `git show U11` 看了这个提交：它没有改 `bootstrap.ts`，也没有新增 `mouse.ts`；这个里程碑真实引入的是 `tui/src/libs/terminal/` 下的终端控制 helper、对应 Vitest 测试，以及产品版本解析的 build-time fallback。

![U11 终端控制模块概览](../../images/u11-terminal-control/terminal-control-overview.png)

## 这个提交实际新增了什么

U11 新增了四类终端能力：

- OSC 2 窗口标题：[`windowTitle.ts`](https://github.com/kefeiqian/KQode/blob/97a9a65ddb34dce8404927ea0cd26f6860b2ed00/tui/src/libs/terminal/windowTitle.ts)。
- OSC 11 / OSC 111 终端背景设置与恢复：[`terminalBackground.ts`](https://github.com/kefeiqian/KQode/blob/97a9a65ddb34dce8404927ea0cd26f6860b2ed00/tui/src/libs/terminal/terminalBackground.ts)。
- DEC private mode 1049 备用屏幕进入与退出：[`alternateScreen.ts`](https://github.com/kefeiqian/KQode/blob/97a9a65ddb34dce8404927ea0cd26f6860b2ed00/tui/src/libs/terminal/alternateScreen.ts)。
- Truecolor ANSI 前景色和可见长度计算：[`ansiColor.ts`](https://github.com/kefeiqian/KQode/blob/97a9a65ddb34dce8404927ea0cd26f6860b2ed00/tui/src/libs/terminal/ansiColor.ts)。

同时，[`productMetadata.ts`](https://github.com/kefeiqian/KQode/blob/97a9a65ddb34dce8404927ea0cd26f6860b2ed00/tui/src/libs/product/productMetadata.ts) 新增 `resolveProductVersion`，让 packaged build 可以通过环境变量注入版本，而 source mode 仍然从 Cargo manifest 读取版本。

```ts
export function resolveProductVersion(options: { repoRoot?: string }): string {
  const injected = process.env.KQODE_VERSION;
  if (injected !== undefined && injected.length > 0) {
    return injected;
  }
  if (options.repoRoot === undefined) {
    throw new Error('product version requires a repo root when no build-time version is injected');
  }
  return readProductVersion(options.repoRoot);
}
```

## 为什么先做终端 helper

窗口标题、背景、备用屏幕这类能力本质上都是往 `stdout` 写控制序列。如果直接在 React component 里写，短期看更快，但后面会带来三个问题：测试会污染 captured output；退出路径会分散；后续 mouse tracking、focus event 或 bracketed paste 的语义边界会越来越模糊。

所以 U11 把每个控制点都拆成纯 builder + `TTY` 写入函数。纯 builder 方便断言 exact bytes；写入函数负责 `stream.isTTY` guard。这个结构让终端控制变成基础设施，而不是 UI 组件里的散落副作用。

## 这不是完整终端 runtime

U11 仍然是一个小步提交。它没有实现鼠标追踪，也没有把这些 helper 全部接入启动生命周期；它只是把后续接线需要的安全原语先放好。这样做的原因很现实：一旦 control sequence 本身和测试策略稳定，后面的 bootstrap 接线就可以只讨论生命周期顺序，而不用同时争论 escape bytes 是否正确。
