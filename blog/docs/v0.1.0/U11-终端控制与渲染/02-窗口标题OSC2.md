U11 新增 [`windowTitle.ts`](https://github.com/kefeiqian/KQode/blob/97a9a65ddb34dce8404927ea0cd26f6860b2ed00/tui/src/libs/terminal/windowTitle.ts)，把终端窗口标题封装成独立 helper。这里使用的是 OSC 2：`ESC ] 2 ; <title> BEL`。它不是 UI 内容的一部分，而是终端 emulator 的窗口 metadata，所以单独放在 `terminal` library 里更合适。

```ts
const SET_WINDOW_TITLE_PREFIX = '\u001B]2;';
const SET_WINDOW_TITLE_SUFFIX = '\u0007';

/** Builds the OSC 2 escape sequence that sets the terminal window title to `title`. */
export function buildWindowTitleSequence(title: string): string {
  return `${SET_WINDOW_TITLE_PREFIX}${title}${SET_WINDOW_TITLE_SUFFIX}`;
}

/** Formats the terminal window title, e.g. `KQode v0.1.0`. */
export function formatWindowTitle(productName: string, productVersion: string): string {
  return `${productName} v${productVersion}`;
}
```

## 为什么标题里要带版本

窗口标题选择 `KQode v0.1.0`，而不是只写 `KQode`。原因是开发阶段经常同时运行 source build、packaged build 和旧版本回归测试。标题带版本可以让用户在多个终端 tab 中快速确认自己操作的是哪一个 build，尤其是后面 release smoke test 时，`kqode --version` 和窗口标题应该指向同一个产品版本。

这也是 U11 同时补 `resolveProductVersion` 的原因。source mode 可以继续从 Cargo manifest 读版本；packaged mode 则通过 build-time env 注入版本，避免发布后的可执行文件还依赖仓库根目录。

## 为什么必须有 `TTY` 守卫

窗口标题 sequence 只应该写给真实终端。U11 的写入函数第一件事就是检查 `stream.isTTY`：

```ts
export function setTerminalWindowTitle(
  productName: string,
  productVersion: string,
  stream: NodeJS.WriteStream = process.stdout
): void {
  if (!stream.isTTY) {
    return;
  }

  stream.write(buildWindowTitleSequence(formatWindowTitle(productName, productVersion)));
}
```

这个 guard 的价值在测试和 pipe 场景里最明显。如果用户执行 `kqode > out.txt`，或者测试框架捕获 `stdout`，我们不希望文件里出现 `\u001B]2;...\u0007`。这类不可见字符会让快照难读、diff 难查，也可能影响下游工具解析。

## 为什么 reset 用空 OSC 2

U11 还提供了 `resetTerminalWindowTitle`：

```ts
export function resetTerminalWindowTitle(stream: NodeJS.WriteStream = process.stdout): void {
  if (!stream.isTTY) {
    return;
  }

  stream.write(buildWindowTitleSequence(''));
}
```

这里没有尝试读取用户原来的窗口标题。终端没有一个简单、跨平台、无副作用的“同步读取标题”接口；强行做会引入异步查询、超时和兼容性问题。用空 OSC 2 清理标题是更保守的选择：它不能完美恢复原始标题，但能保证 KQode 设置的标题不会长期残留。

对应测试在 [`windowTitle.test.ts`](https://github.com/kefeiqian/KQode/blob/97a9a65ddb34dce8404927ea0cd26f6860b2ed00/tui/src/libs/terminal/__tests__/windowTitle.test.ts) 里直接断言 exact sequence：

```ts
expect(buildWindowTitleSequence('KQode v0.1.0')).toBe('\u001B]2;KQode v0.1.0\u0007');
expect(formatWindowTitle('KQode', '0.1.0')).toBe('KQode v0.1.0');
```

这类测试看起来很“低层”，但它正好保护了终端控制代码最容易出错的部分：一个少写的 `BEL`、一个错误的 OSC 编号，都会让终端行为变得不可预测。
