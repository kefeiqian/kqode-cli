U13 很小，但它是 `v0.1.0` 的收口提交：tag `U13` 和 tag `v0.1.0` 指向同一个 commit [`a1953dbd`](https://github.com/kefeiqian/KQode/commit/a1953dbddf0b1902e8dfe46220e7aeb33cf829cb)。从这里开始，KQode 有了跨平台 release asset pipeline，而不是只有本地打包脚本。

## 关键决策与为什么

- `release.yml` 使用 native runner matrix。为什么：U12 的打包脚本只负责当前 host；真实跨平台 artifacts 应该在对应平台构建，减少 cross compile 和 Bun standalone 差异。
- `workflow_dispatch` 做 dry run，`v*` tag 才发布。为什么：维护者需要验证流水线而不误发公开 release；版本 tag 是发布意图的清晰信号。
- required target fail closed，optional target best-effort。为什么：缺少 Linux / macOS / Windows x64 这类核心 artifact 必须阻断 release；Windows ARM64 仍可尝试但不阻塞第一版。
- 汇总前先校验每个 `.sha256`。为什么：Release 页面上的 `checksums.txt` 必须来自已验证 archive，不能只是简单拼接未知文件。
- npm publish 独立 workflow。为什么：npm package 安装时依赖 GitHub Release assets；把 npm 发布从 release build 中拆开，可以确保顺序，也避免长期 `NPM_TOKEN`。

```yaml
concurrency:
  group: release-${{ github.ref }}
  cancel-in-progress: false
```

这个 concurrency 设置也很关键：同一个 ref 的 release 不应并发互相覆盖；但已经开始的发布也不应该被新的 run 取消，避免半途留下不完整 artifacts。

## 明确推迟的事

U13 没有自动提交 Homebrew tap 或 winget manifest，没有做 code signing / notarization，也没有实现 auto-update。这些都需要额外凭据、平台账号和安全策略。当前提交先完成 GitHub Release assets、checksum、attestation 和 npm Trusted Publishing 的骨架，是更适合 `v0.1.0` 的范围。

## U11 到 U13 的连贯性

U11 让 TUI 更像一个真实终端应用；U12 让它有可下载、可校验、可被 package manager 消费的 artifacts；U13 让这些 artifacts 能由 tag 自动构建并上传。三步合起来，KQode `v0.1.0` 从本地工程走到了可发布产品形态。
