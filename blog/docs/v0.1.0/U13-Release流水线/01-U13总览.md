U13 对应 commit [`a1953dbd`](https://github.com/kefeiqian/KQode/commit/a1953dbddf0b1902e8dfe46220e7aeb33cf829cb)，主题是 `ci(release): add the GitHub Release asset pipeline`。这个 commit 同时也是 tag `v0.1.0` 指向的 commit，也就是说 U13 完成了 KQode `v0.1.0` 的 release 流水线闭环。

这个提交很小，只新增两个 GitHub Actions workflow：[`release.yml`](https://github.com/kefeiqian/KQode/blob/a1953dbddf0b1902e8dfe46220e7aeb33cf829cb/.github/workflows/release.yml) 和 [`npm-publish.yml`](https://github.com/kefeiqian/KQode/blob/a1953dbddf0b1902e8dfe46220e7aeb33cf829cb/.github/workflows/npm-publish.yml)。但它的意义很大：U12 已经定义了如何在单个 host 上生成 release artifact；U13 把这件事扩展到跨平台 native runner matrix，并在 version tag 上创建或更新 GitHub Release。

![U13 Release 流水线概览](../../images/u13-release-pipeline/release-pipeline-overview.png)

## U13 的分工

`release.yml` 负责构建和上传 GitHub Release assets：

```yaml
on:
  push:
    tags:
      - "v*"
  workflow_dispatch: {}
```

`npm-publish.yml` 负责发布单个 npm package，并刻意不强行塞进 `release.yml`：

```yaml
name: Publish npm

on:
  workflow_dispatch:
    inputs:
      tag:
        description: "Release tag to publish, e.g. v0.1.0"
        required: true
        type: string
  release:
    types: [published]
```

这种拆分的原因是依赖顺序不同。npm package 的 postinstall 会下载 GitHub Release 上的 archive，所以 npm 发布必须发生在 release assets 上传之后。把 npm publish 作为独立 workflow，可以避免 GitHub Release 还没准备好时 npm 包已经可见。

## 为什么说 U13 完成 `v0.1.0`

U11 解决终端基础体验，U12 准备分发产物，U13 则让版本 tag 触发实际 release。三者合起来，`v0.1.0` 不再只是源码 tag，而是一个可以生成多平台 artifacts、校验 checksum、上传 GitHub Release，并可通过 npm Trusted Publishing 发布 CLI wrapper 的版本。

U13 没有引入新的应用功能；它引入的是发布能力。对于开发者工具来说，这同样是产品能力：用户拿不到可安装产物，再完整的本地功能也很难成为 release。
