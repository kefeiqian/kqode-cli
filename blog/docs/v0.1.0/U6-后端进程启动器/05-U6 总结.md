title: 5. U6 总结
---

U6 把 KQode TUI 到 Rust 后端之间的第一段桥接补上了：不是 JSON-RPC client，而是一个 guarded source backend process launcher。它负责从可信源码构建后端、在用户 workspace cwd 里启动后端、收敛环境变量、暴露 stdio handles，并在错误或退出时清理 process tree。

## 交付物

- [`backendBuild.ts`](https://github.com/kefeiqian/KQode/blob/29eaf0a236e7ec16f2fc39f9aeea709f74e1f0dd/tui/src/backend/process/backendBuild.ts)：source-mode `cargo build --bin kqode`，平台相关二进制路径解析，构建 timeout 和 capped stderr。
- [`backendProcess.ts`](https://github.com/kefeiqian/KQode/blob/29eaf0a236e7ec16f2fc39f9aeea709f74e1f0dd/tui/src/backend/process/backendProcess.ts)：`spawnBackend` 和 `launchSourceBackend`，对外只暴露 `LaunchedBackend`。
- [`processEnv.ts`](https://github.com/kefeiqian/KQode/blob/29eaf0a236e7ec16f2fc39f9aeea709f74e1f0dd/tui/src/backend/process/processEnv.ts)：Windows / Unix allowlist 环境构造。
- [`processUtils.ts`](https://github.com/kefeiqian/KQode/blob/29eaf0a236e7ec16f2fc39f9aeea709f74e1f0dd/tui/src/backend/process/processUtils.ts)：`killProcessTree` 和 `CappedBuffer`。
- 三组测试：路径解析与构建错误、环境收敛、真实 source-mode 后端启动。

## 关键决策与原因

### 1. build 从 `repoRoot`，run 从 `workspaceCwd`

```ts
await buildBackend({ repoRoot, timeoutMs: buildTimeoutMs });
return await spawnBackend({
  binaryPath: resolveBackendBinaryPath(repoRoot),
  workspaceCwd
});
```

**为什么：** KQode 后端源码属于可信仓库，用户项目只是被操作对象。构建从 `repoRoot` 发生，可以避免用户 workspace 的 Cargo 配置影响 KQode 自己；运行从 `workspaceCwd` 发生，则保证后端看到的是用户项目路径。

### 2. 环境变量使用 allowlist

**为什么：** provider token、registry credential 和 cloud secret 不应该默认进入后端进程。denylist 很容易漏掉组织内部变量名；allowlist 虽然更保守，但更符合 agent harness 的安全边界。

### 3. `spawn` 只证明进程启动，不证明协议 ready

**为什么：** 一个进程可以启动成功但永远不输出合法 JSON-RPC。U6 只负责启动和 stdio pipe，真正的 readiness 交给 U7 的 JSON-RPC client 处理，避免 process 层和协议层耦合。

### 4. timeout 后 kill process tree

**为什么：** 只杀父进程可能留下 `rustc` 或后端子进程。source-mode 开发链路如果经常留下孤儿进程，会让后续测试和本地开发变得不可预测。

## 暂缓事项

- U6 没有实现 JSON-RPC request / response 关联；这是 U7 的范围。
- U6 没有把 stderr 诊断接到 UI 展示；它只先保留 bounded diagnostics。
- U6 没有处理 packaged backend；这个里程碑只覆盖 source-mode 开发路径。
- U6 没有实现 session restore 或请求重放；launcher 不应该知道业务语义。
