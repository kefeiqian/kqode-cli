title: 4. 生命周期错误处理与测试
---

U6 的 launcher 不只负责 happy path。它要回答三个失败问题：构建不能开始怎么办，构建卡住怎么办，后端二进制不能启动怎么办。把这些错误在 launcher 层分清楚，U7 的 JSON-RPC client 才能继续区分启动错误、协议错误和请求超时。

![U6 后端进程 launcher 错误路径测试](../../images/u6-backend-launcher/launcher-error-path-tests.png)

## 构建超时是 timeout，不是 launch

[`buildBackend`](https://github.com/kefeiqian/KQode/blob/29eaf0a236e7ec16f2fc39f9aeea709f74e1f0dd/tui/src/backend/process/backendBuild.ts) 使用 timer 包住 build 进程。超时后先清理 process tree，再抛出 `BackendErrorKind.Timeout`：

```ts
const timer = setTimeout(() => {
  settle(() => {
    killProcessTree(build.pid);
    reject(
      new BackendClientError(
        BackendErrorKind.Timeout,
        `backend build timed out after ${timeoutMs}ms`
      )
    );
  });
}, timeoutMs);
```

为什么 timeout 要单独建模？因为“命令不存在”和“命令存在但挂住”对用户的下一步动作不同。前者通常是路径、工具链或权限问题；后者可能是依赖下载、锁竞争、编译卡死或平台问题。把它们都叫 launch failed，会让 UI 后续无法给出准确提示。

## 非零退出保留 capped stderr

构建进程正常退出但 exit code 非 0 时，U6 把 stderr 放进 launch error：

```ts
build.once('close', (code) => {
  settle(() => {
    if (code === 0) {
      resolve();
      return;
    }
    reject(new BackendClientError(BackendErrorKind.Launch, buildFailureMessage(code, stderr)));
  });
});
```

```ts
function buildFailureMessage(code: number | null, stderr: CappedBuffer): string {
  const detail = stderr.toString().trim();
  const base = `backend build exited with code ${code ?? 'null'}`;
  return detail.length === 0 ? base : `${base}: ${detail}`;
}
```

这里没有把 stderr 的“存在”当成失败。很多编译器和工具会把 warning、进度或诊断写到 stderr。真正的失败信号是 exit code，而不是 stderr 是否为空。

## spawn 成功不等于 backend ready

[`spawnBackend`](https://github.com/kefeiqian/KQode/blob/29eaf0a236e7ec16f2fc39f9aeea709f74e1f0dd/tui/src/backend/process/backendProcess.ts) 只等待 Node 的 `spawn` 事件，并确认 stdio pipes 存在：

```ts
await waitForSpawn(child);

const { stdin, stdout, stderr } = child;
if (stdin === null || stdout === null || stderr === null) {
  killProcessTree(child.pid);
  throw new BackendClientError(BackendErrorKind.Launch, 'backend process is missing stdio pipes');
}
```

这也是一个有意的分层：OS 已经启动进程，只能说明 executable 可以运行；它是否真的会说 JSON-RPC，要到 U7 的 readiness gate 才能判断。U6 不提前解析协议，是为了让 process 层保持传输无关。

## 集成测试验证 workspace cwd

U6 的集成测试没有只在 repo root 跑一次，而是额外用 dummy React fixture 验证 `workspaceCwd`：

```ts
it(
  'preserves a distinct workspace cwd such as the dummy React fixture path',
  async () => {
    const workspaceCwd = path.join(repoRoot, 'tests', 'fixtures', 'dummy-react-app');
    const backend = await launchSourceBackend({ repoRoot, workspaceCwd });
    try {
      expect(await ackThroughLauncher(backend, 'café ☕')).toBe('café ☕');
    } finally {
      backend.dispose();
    }
  },
  INTEGRATION_TIMEOUT_MS
);
```

为什么这个测试重要？因为 Coding Agent 的后端应该服务“用户当前项目”，不是服务 `tui/` package。即使 source-mode build 从 KQode 的 `repoRoot` 发生，后端运行时 cwd 也必须是用户 workspace。这个边界一旦错了，后续文件读取、Git 状态、shell 工具和路径显示都会错。

## 缺失 executable 的失败路径

测试还覆盖了 spawn 缺失二进制：

```ts
describe('spawnBackend', () => {
  it('rejects with a launch error when the backend executable is missing', async () => {
    await expect(
      spawnBackend({
        binaryPath: path.join(process.cwd(), 'no-such-kqode-backend-binary'),
        workspaceCwd: process.cwd()
      })
    ).rejects.toMatchObject({ kind: BackendErrorKind.Launch });
  });
});
```

这类测试看似普通，但它保证了 launcher 不会把 Node 的原始 `ENOENT` 直接泄漏给 UI。后续 UI 可以基于 `BackendErrorKind.Launch` 统一渲染“后端启动失败”，同时保留底层 cause 做诊断。
