title: 2. 解析后端构建与启动路径
---

U6 的第一层问题是：TUI 应该怎么找到 Rust 后端？最简单的做法是直接假设 `target/debug/kqode` 已经存在。但这个假设对 fresh checkout 不成立，也会把“开发者是否手动跑过 cargo build”变成 UI 启动是否成功的隐性前置条件。

所以 U6 选择 source-mode launcher：从可信仓库根目录先构建，再解析后端二进制路径。

## 平台相关的二进制路径

[`resolveBackendBinaryPath`](https://github.com/kefeiqian/KQode/blob/29eaf0a236e7ec16f2fc39f9aeea709f74e1f0dd/tui/src/backend/process/backendBuild.ts) 只做一件事：按平台拼出 debug profile 的可执行文件名。

```ts
export function resolveBackendBinaryPath(repoRoot: string, platform = process.platform): string {
  const fileName = platform === 'win32' ? `${CARGO_BINARY_NAME}.exe` : CARGO_BINARY_NAME;
  return path.join(repoRoot, 'target', 'debug', fileName);
}
```

为什么要把它单独拆出来？因为 Windows 的 `.exe` 后缀不是展示细节，而是启动语义的一部分。把它集中在一个 helper 里，测试就能直接覆盖 `win32` 和 `linux` 两种命名，而不是让平台分支散落在更复杂的 launcher 逻辑里。

对应测试也非常直接：

```ts
describe('resolveBackendBinaryPath', () => {
  it('uses platform-correct debug executable naming', () => {
    expect(resolveBackendBinaryPath('/repo', 'win32')).toBe(
      path.join('/repo', 'target', 'debug', 'kqode.exe')
    );
    expect(resolveBackendBinaryPath('/repo', 'linux')).toBe(
      path.join('/repo', 'target', 'debug', 'kqode')
    );
  });
});
```

## 为什么 build 从 `repoRoot` 发生

[`buildBackend`](https://github.com/kefeiqian/KQode/blob/29eaf0a236e7ec16f2fc39f9aeea709f74e1f0dd/tui/src/backend/process/backendBuild.ts) 的默认命令是 `cargo build --bin kqode`，并且 `cwd` 固定为 `repoRoot`：

```ts
export async function buildBackend({
  repoRoot,
  env = buildHardenedEnv({ includeCargo: true }),
  timeoutMs = DEFAULT_BUILD_TIMEOUT_MS,
  command = CARGO_COMMAND,
  args = ['build', '--bin', CARGO_BINARY_NAME]
}: BuildBackendOptions): Promise<void> {
  const build = spawn(command, [...args], {
    cwd: repoRoot,
    env,
    shell: false,
    stdio: ['ignore', 'ignore', 'pipe'],
    windowsHide: true
  });
```

这里拒绝了两个替代方案：

1. **从 `tui/` package cwd 构建。** 这会把 package script 的执行目录和 Rust workspace 绑定起来。后续 TUI 可能从 fixture、IDE、packaged binary 或不同 shell 启动，不能依赖 `process.cwd()` 正好是仓库根目录。
2. **从用户 workspace cwd 构建。** Coding Agent 操作的是用户项目，用户项目可能有自己的 `.cargo/config.toml`、PATH 注入或其他构建配置。KQode 后端必须从 KQode 自己的可信 `repoRoot` 构建，而不是从被操作项目里构建。

## 为什么构建 stdout 被忽略，只保留 stderr

构建阶段的 stdout 对 TUI 不是协议数据。后续 JSON-RPC 走后端进程的 stdout，如果这里不区分构建进程和服务进程，很容易把日志、构建进度和协议帧混在一起。U6 因此让 build 的 `stdio` 是：stdin ignore、stdout ignore、stderr pipe。

```ts
const stderr = new CappedBuffer(BUILD_STDERR_CAP_BYTES);
build.stderr?.on('data', (chunk: Buffer) => stderr.append(chunk));
```

只保留 stderr 的原因是：构建失败时需要给开发者看到关键诊断，但不能无上限收集整段编译输出。这个边界在 U6 还没有接 UI 展示，但 launcher 已经先把“有用诊断”和“输出上限”两个需求放进模型里。

## 这一步为 U7 留出的接口

U6 没有在这里做 JSON-RPC。它只返回 `LaunchedBackend`，也就是后续 client 需要的 transport-neutral handles：

```ts
export type LaunchedBackend = {
  readonly pid: number | undefined;
  readonly stdin: Writable;
  readonly stdout: Readable;
  readonly stderr: Readable;
  onExit(listener: (exit: BackendExit) => void): void;
  dispose(): void;
};
```

这个取舍很关键：launcher 不知道 message submit、不知道 request id、不知道 UI 状态；它只负责“安全地产生一个可通信的进程”。这样 U7 可以把 JSON-RPC client 独立接上，而不会反向污染 process launcher。
