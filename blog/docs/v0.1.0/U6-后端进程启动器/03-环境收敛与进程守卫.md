title: 3. 环境收敛与进程守卫
---

启动本地进程时，最容易偷懒的写法是把 `process.env` 原样传给 child process。U6 刻意没有这么做。原因是 KQode 未来会处理用户代码、模型供应商、工具执行和审批策略；如果一开始就让后端继承所有环境变量，provider key、registry token、SSH agent 句柄和 CI secret 都可能被带进不需要它们的进程边界。

## allowlist，而不是 denylist

[`buildHardenedEnv`](https://github.com/kefeiqian/KQode/blob/29eaf0a236e7ec16f2fc39f9aeea709f74e1f0dd/tui/src/backend/process/processEnv.ts) 使用平台 allowlist：

```ts
const WINDOWS_ALLOWLIST = [
  'PATH',
  'PATHEXT',
  'SYSTEMROOT',
  'WINDIR',
  'TEMP',
  'TMP',
  'USERPROFILE',
  'HOMEDRIVE',
  'HOMEPATH',
  'HOME'
];

const UNIX_ALLOWLIST = ['PATH', 'HOME', 'TMPDIR', 'TERM', 'COLORTERM', 'LANG', 'LC_ALL', 'LC_CTYPE'];

const CARGO_ALLOWLIST = ['CARGO_HOME', 'RUSTUP_HOME', 'RUSTUP_TOOLCHAIN'];
```

为什么不用 denylist？因为 secret 命名没有统一标准。`GITHUB_TOKEN`、`AWS_SECRET_ACCESS_KEY`、`AZURE_CLIENT_SECRET` 只是常见例子；实际团队还可能有私有变量名。denylist 总会漏。allowlist 的思路相反：只把启动进程确实需要的系统变量传下去。

## Cargo 变量只在 build 阶段打开

同一个 helper 有 `includeCargo` 开关：

```ts
export function buildHardenedEnv({
  includeCargo = false,
  platform = process.platform,
  source = process.env
}: HardenedEnvOptions = {}): NodeJS.ProcessEnv {
  const baseAllowlist = platform === 'win32' ? WINDOWS_ALLOWLIST : UNIX_ALLOWLIST;
  const allowed = new Set(
    [...baseAllowlist, ...(includeCargo ? CARGO_ALLOWLIST : [])].map((name) => name.toUpperCase())
  );
```

构建 Rust 后端时需要 `CARGO_HOME`、`RUSTUP_HOME` 或 `RUSTUP_TOOLCHAIN` 这类变量，否则本地工具链可能无法定位。但运行已经构建好的后端时，这些变量不再是必要输入。U6 因此把 build 和 run 的环境需求分开，避免“为了构建方便”把 Cargo 相关环境永久泄漏到服务进程。

测试明确表达了这个边界：

```ts
it('excludes Cargo variables unless a build needs them', () => {
  expect(buildHardenedEnv({ platform: 'linux', source: unixSource }).CARGO_HOME).toBeUndefined();
  expect(
    buildHardenedEnv({ platform: 'linux', source: unixSource, includeCargo: true }).CARGO_HOME
  ).toBe('/home/dev/.cargo');
});
```

## process-tree cleanup，而不是只 kill 父进程

构建超时或 TUI 退出时，只杀掉父进程不够。`cargo build` 下面可能还有 `rustc` 子进程；未来后端也可能再启动工具进程。U6 用 [`tree-kill`](https://github.com/kefeiqian/KQode/blob/29eaf0a236e7ec16f2fc39f9aeea709f74e1f0dd/tui/src/backend/process/processUtils.ts) 做 best-effort process-tree 清理：

```ts
export function killProcessTree(pid: number | undefined): void {
  if (pid === undefined) {
    return;
  }

  treeKill(pid, 'SIGKILL', () => {
    // Best-effort cleanup: a missing process is already gone.
  });
}
```

这里的“best-effort”很现实：进程可能已经自然退出，或者 OS 回收顺序比 cleanup 回调更快。launcher 不应该因为“要清理的进程已经不存在”再制造一个新的 fatal error。

## capped buffer 的原因

[`CappedBuffer`](https://github.com/kefeiqian/KQode/blob/29eaf0a236e7ec16f2fc39f9aeea709f74e1f0dd/tui/src/backend/process/processUtils.ts) 的目标不是做完整日志系统，而是在错误消息里保留有限诊断：

```ts
export class CappedBuffer {
  private readonly chunks: Buffer[] = [];
  private size = 0;

  constructor(private readonly capBytes: number) {}

  append(chunk: Buffer): void {
    if (this.size >= this.capBytes) {
      return;
    }

    const remaining = this.capBytes - this.size;
    const slice = chunk.length > remaining ? chunk.subarray(0, remaining) : chunk;
    this.chunks.push(slice);
    this.size += slice.length;
  }
```

不用无限 buffer 的原因有两个：第一，构建失败可能输出大量诊断，TUI 不应该因为错误路径吃掉太多内存；第二，用户真正需要的是“为什么失败”的开头线索，而不是完整编译日志。完整日志后续可以通过 trace 或专门日志面板补，不应该塞进 launcher error message。
