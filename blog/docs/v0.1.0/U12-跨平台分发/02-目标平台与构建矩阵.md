U12 在 [`resolve.cjs`](https://github.com/kefeiqian/KQode/blob/a667e95e4ba1832be0b94fb4fb027cae2f5438ad/packaging/npm/kqode/lib/resolve.cjs) 里定义了 npm package 能识别的 host target。这个文件是纯函数模块，不做网络和文件 I/O，专门负责把 Node 的 `process.platform` / `process.arch` 映射到 release asset 名称。

```js
const SUPPORTED_TARGETS = [
  'darwin-arm64',
  'linux-arm64',
  'linux-x64',
  'win32-arm64',
  'win32-x64'
];

/** Maps Node's `process.platform` to the release-archive OS segment. */
const RELEASE_OS = { darwin: 'darwin', linux: 'linux', win32: 'windows' };
```

## 为什么 release 名称不用 Node 平台名 `win32`

Node 的 Windows 平台名是 `win32`，但 release asset 使用 `windows`：

```js
function releaseTargetName(platform, arch) {
  const override = ASSET_OVERRIDES[platformKey(platform, arch)];
  const assetOs = override ? override.os : RELEASE_OS[platform];
  const assetArch = override ? override.arch : arch;
  return `kqode-${assetOs}-${assetArch}`;
}
```

原因是 artifact 名称是用户可见的下载协议，不应该泄漏 Node runtime 的历史命名。`kqode-windows-x64.zip` 比 `kqode-win32-x64.zip` 更符合 GitHub Release、Homebrew / winget 文档和用户直觉。U12 在 mapping 层吸收这种差异，让安装器内部可以继续使用 Node 原生 platform 值。

## 为什么 Windows ARM64 映射到 x64 asset

U12 对 `win32-arm64` 做了 override：

```js
const ASSET_OVERRIDES = { 'win32-arm64': { os: 'windows', arch: 'x64' } };
```

对应测试在 [`resolve.test.cjs`](https://github.com/kefeiqian/KQode/blob/a667e95e4ba1832be0b94fb4fb027cae2f5438ad/packaging/npm/kqode/test/resolve.test.cjs) 里写得很直接：

```js
test('win32-arm64 falls back to the windows-x64 asset (x64 emulation)', () => {
  assert.equal(releaseTargetName('win32', 'arm64'), 'kqode-windows-x64');
  assert.ok(isSupported('win32', 'arm64'));
});
```

这不是说 Windows ARM64 没有用户，而是 U12 阶段没有 native Windows ARM64 build。Windows 11 on ARM 可以通过 x64 emulation 运行 x64 executable，所以 npm 层把它视为 supported，并下载 `kqode-windows-x64.zip`。这个选择的好处是先覆盖用户安装路径；代价是性能和 native 体验不如真正的 ARM64 build。

## 为什么 Intel macOS 不在矩阵里

测试也明确断言 `darwin-x64` 不支持：

```js
assert.ok(!isSupported('darwin', 'x64'));
```

这是一个发布范围选择。KQode v0.1.0 的第一批目标优先覆盖 Apple Silicon macOS、Linux x64 / ARM64、Windows x64，并通过 emulation 覆盖 Windows ARM64。Intel macOS 被推迟，原因通常不是代码不可能运行，而是每多一个 target 就增加 CI 时间、artifact 数量、验证成本和渠道注册复杂度。
