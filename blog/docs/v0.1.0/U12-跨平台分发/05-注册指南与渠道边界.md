U12 新增 [`kqode_distribution_registration.md`](https://github.com/kefeiqian/KQode/blob/a667e95e4ba1832be0b94fb4fb027cae2f5438ad/docs/release/kqode_distribution_registration.md)。这不是普通 README，而是把分发渠道边界写成维护者手册：GitHub Release 负责原始 artifacts，npm 负责 thin installer，Homebrew 和 winget 负责引用同一批 release URL 和 checksum。

```md
Every channel distributes the *same* per-platform standalone executable. Package
managers are thin installers that select or download that executable; none of
them build KQode from source or require Cargo, Node, or npm at runtime (except
npm, which is itself the installer for the npm channel).
```

![分发渠道共用同一批 release artifacts](../../images/u12-cross-platform-distribution/channel-boundaries.png)

## 为什么所有渠道都消费同一批 executable

如果 npm、Homebrew、winget 各自构建 KQode，发布问题会变得很难追踪：同一个版本号可能对应不同编译环境、不同依赖、不同 commit。U12 明确要求所有渠道分发同一批 GitHub Release assets，这样 `v0.1.0` 的 `kqode-linux-x64.tar.gz` 就是唯一真相。

这个选择也让验证链路更简单：每个渠道只需要 pin GitHub Release URL 和 SHA-256，而不是重新证明自己的构建过程。

## 为什么 Homebrew 和 winget 先手动注册

文档的 Scope 里明确推迟了部分自动化：

```md
Deferred (not covered here, intentionally out of scope for this slice):

- Homebrew tap submission and winget submission from CI.
- Code signing, notarization, and auto-update.
```

这是典型的 milestone 取舍。v0.1.0 首要目标是把可下载 artifact 和 checksum 做出来，并有清晰手册让 maintainer 完成注册。自动提交 Homebrew tap、自动 PR 到 `winget-pkgs`、签名、公证、自动更新都需要额外权限和安全审查；把它们放进同一个提交会让 release pipeline 难以 review。

## 为什么版本号必须等于 tag

文档要求产品版本来自根目录 `Cargo.toml`，并且等于 release tag：

```bash
cargo xtask set-version 0.2.0

git commit -am "chore: release v0.2.0"
git tag v0.2.0 && git push origin v0.2.0
```

这是为了避免三种版本漂移：`kqode --version` 显示一个版本，npm package 是另一个版本，GitHub Release tag 又是第三个版本。U12 的产品版本注入、npm package version 和 release asset URL 都围绕 `v<version>` 展开，所以 tag / manifest 一致性是发布正确性的前提。

## 为什么文档提前写 Trusted Publishing

npm 部分明确推荐 Trusted Publishing，而不是长期 `NPM_TOKEN`：

```md
No `NPM_TOKEN` secret is needed for automated publishing; OIDC replaces it.
```

这属于安全边界设计。长期 token 容易泄漏、过期和权限过大；OIDC trusted publishing 把“哪个 GitHub workflow 可以发布哪个 npm package”交给 npm 和 GitHub 的身份联邦。U12 先写下这个方向，U13 的 `npm-publish.yml` 才能按这个安全模型实现。
