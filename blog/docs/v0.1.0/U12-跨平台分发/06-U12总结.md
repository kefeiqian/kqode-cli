title: 6. U12 总结
---

U12 把 KQode 从“本地可运行”推进到“有分发形状”。它没有发布 release，也没有跑 GitHub Actions；它做的是发布前一层：定义 artifact、打包命令、npm 安装器和渠道注册协议。

## 关键决策与为什么

- `cargo xtask package-release` 先调用已有 `package`，再运行 Bun 打包脚本。为什么：Rust `xtask` 保持统一入口，TypeScript script 复用 npm mapping 和 Node 文件处理能力。
- 本地命令只打当前 host。为什么：standalone executable 和 Bun 产物最好在 native runner 上构建；跨平台矩阵应该交给 CI，而不是复杂 cross compile。
- Artifact 命名统一为 `kqode-<os>-<arch>`。为什么：同一套名字被 GitHub Release、npm、Homebrew、winget 共同消费，减少渠道间转换。
- npm 使用单 package thin launcher。为什么：维护一个 `@kqode/kqode-cli` 比维护多个 per-platform package 更适合 v0.1.0；代价是安装或首次运行需要下载 GitHub Release asset。
- 所有下载都校验 SHA-256。为什么：发布二进制必须至少能发现 asset 损坏或错配，后续再叠加 attestations、签名和公证。
- Homebrew、winget 自动提交推迟。为什么：它们需要额外账号、权限和 registry 流程；U12 先把可注册的 artifact 与手册准备好。

```js
function releaseBaseUrl(version) {
  return `https://github.com/${REPO}/releases/download/v${version}`;
}
```

## 明确推迟的事

U12 推迟了 CI release pipeline，这正是 U13 要做的内容。它也推迟了 native Windows ARM64 binary，用 x64 emulation 先覆盖安装路径。签名、公证、自动更新、Homebrew tap PR 和 winget PR 都没有进入这个提交，因为它们属于发布治理和供应链增强，不应阻塞第一版 artifact 形状。

## 这个单元留下的基础

到 U12 结束时，KQode 已经有了一个清楚的分发协议：native runner 产出 archive + checksum，GitHub Release 托管这些文件，npm / Homebrew / winget 都消费它们。U13 只需要把这个协议自动化成 workflow。
