U2 的 `HomeScreen.tsx`（近 190 行）和 `PromptComposer.tsx`（约 300 行）都超过或逼近 `tui/AGENTS.md`“文件保持在 200 行以内”的约定。U3 把它们拆成 colocated 文件夹：一个 `index` 做编排，其余各是聚焦模块。

## 为什么要拆

200 行不是拍脑袋的数字，而是一条“可读性刹车”。一个文件一旦同时装下状态、布局、输入处理、光标数学、渲染，改任何一处都得先在脑子里加载全部上下文。拆分的目标不是行数达标，而是**让每个文件只有一个职责**：读它就知道它管什么，改它不用担心碰到别的。Jotai 迁移（第 2 篇）把状态抽走后，组件本身也就有了干净拆分的前提。

## HomeScreen：拆成状态边界 + 视图

`HomeScreen/` 现在是两层：

- `HomeScreen/index.tsx`：只做状态边界——建 Jotai store、注入 config、`Provider` 包裹（见第 2 篇）。
- `HomeScreen/HomeScreenView.tsx`：负责纵向排版和滚动按键，把五块组合起来。

`HomeScreenView` 里每一块都是一个只读若干 atom 的小包装组件：

```text
HomeScreenView
├─ Header
├─ HomeBody     → BodyPane      （读 displayedBodyEntries / layout / scroll 原子）
├─ HomeCwd      → CwdLine       （读 config 里的 cwd / gitStatusLabel）
├─ HomeComposer → PromptComposer（读 composerTop / layout，写 submitPrompt）
└─ HomeStatus   → StatusBar     （读 modelLabel）
```

这些 `Home*` 包装组件是“atom → props”的适配层：它们从 store 读出叶子组件需要的那几个值，转成普通 props 传下去。好处是**叶子组件（`BodyPane`/`CwdLine`/`StatusBar`）保持纯展示**，不 import 任何 atom，既好测又能复用。

## PromptComposer：一拆为六

U2 的 `PromptComposer.tsx` 把输入处理、可见窗口计算、光标数学、帧渲染全塞在一个文件。U3 把它拆成 `PromptComposer/` 下的六个模块：

| 文件 | 职责 |
| --- | --- |
| `index.tsx` | 编排：读状态、串起输入 hook、算光标、渲染 frame |
| `constants.ts` | `PROMPT_PREFIX` 等常量 |
| `usePromptComposerInput.ts` | `useInput` 键盘处理，dispatch 到 composer 动作 atom |
| `promptTextView.ts` | 文本换行成可见窗口（`formatVisiblePromptView` 等纯函数） |
| `cursorPosition.ts` | 光标坐标数学（`resolveComposerCursorPosition`） |
| `ComposerFrame.tsx` | 把可见文本行、背景、校验错误行渲染出来 |

`index.tsx` 变成一个薄薄的编排层，一眼能看清整条链路：

```tsx
export function PromptComposer({ columns, onSubmit, isActive = true, maxBytes = PROMPT_MAX_BYTES, maxVisibleLines = DEFAULT_COMPOSER_VISIBLE_LINES, cursorTop, onVisibleRowsChange }: PromptComposerProps) {
  const state = useAtomValue(composerStateAtom);
  const composerRef = useRef<DOMElement | null>(null);
  const composerMetrics = useBoxMetrics(composerRef);
  const { setCursorPosition } = useCursor();

  usePromptComposerInput({ isActive, maxBytes, onSubmit, state });

  const inputColumns = Math.max(1, columns - PROMPT_PREFIX.length);
  const visiblePrompt = formatVisiblePromptView(state.text, inputColumns, maxVisibleLines, state.cursorIndex);
  const visibleText = visiblePrompt.text;
  const visibleRows = countVisibleComposerRows(visibleText, state.validationError !== null, true);

  useEffect(() => { onVisibleRowsChange?.(visibleRows); }, [onVisibleRowsChange, visibleRows]);

  if (isActive && composerMetrics.hasMeasured) {
    setCursorPosition(resolveComposerCursorPosition(visibleText, inputColumns, cursorTop ?? composerMetrics.top, visiblePrompt.cursorIndex, true));
  } else {
    setCursorPosition(undefined);
  }

  return (
    <Box ref={composerRef} flexDirection="column">
      <ComposerFrame columns={columns} shouldRenderBackground validationError={state.validationError} visibleTextRows={visibleText.split('\n')} />
    </Box>
  );
}
```

对比 U2 那个 300 行的版本：键盘细节进了 `usePromptComposerInput`、换行细节进了 `promptTextView`、光标数学进了 `cursorPosition`、渲染进了 `ComposerFrame`。`index` 只剩“读状态 → 算可见文本 → 定光标 → 渲染”的骨架。

顺带一提，这一版把 `shouldRenderBackground` 简化成了常量 `true`——U2 那套按色深/`NO_COLOR` 判断的逻辑暂时收敛为“默认开”，把 fallback 判断的复杂度先从这条热路径上移走。

## 输入 hook：usePromptComposerInput

键盘处理被抽成一个 hook，内部 `useInput` 把按键翻译成对 composer 动作 atom 的 `set`（第 2 篇讲过这些动作 atom）。行为和 U2 完全一致：Shift+Enter 换行、Enter 校验后提交、方向键移动、退格删除、可打印字符插入。抽成 hook 的好处是 `index.tsx` 不用被一大段 `useInput` 回调撑大，输入逻辑也能单独测。
