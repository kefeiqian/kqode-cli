---

U7 的核心不是手写 request id map，因为 `vscode-jsonrpc` 已经负责 request / response id correlation。KQode 自己要解决的是更高一层：什么时候启动后端，什么时候复用后端，什么时候判定后端死亡，以及并发 submit 或 eager start 时不要重复 spawn。

## lifecycle state 是显式模型

[`backendClient.ts`](https://github.com/kefeiqian/KQode/blob/7c9d43fa3af1413967813e51709ee6790151e8aa/tui/src/backend/client/backendClient.ts) 用字符串常量表达 lifecycle：

```ts
export const BackendLifecycleState = {
  /** No backend has been launched yet. */
  Idle: 'idle',
  /** A backend launch/connect is in flight. */
  Starting: 'starting',
  /** The backend is connected and accepting requests. */
  Ready: 'ready',
  /** The TUI is disposing the backend on purpose. */
  Closing: 'closing',
  /** The backend exited, crashed, or a fatal transport error occurred. */
  Dead: 'dead'
} as const;
```

为什么不用几个 boolean？因为状态之间有约束。`starting && disposed`、`ready && starting` 这种组合如果用 boolean 表达，很容易出现非法状态。显式 state 让测试可以直接断言 `Idle -> Starting -> Ready -> Dead` 这类转换。

## single-flight startup

`ensureSession` 里有一个 `starting` promise。它保证 eager start 和第一次 submit 同时发生时，不会启动两个 backend：

```ts
const ensureSession = (): Promise<BackendSession> => {
  if (disposed) {
    return Promise.reject(disposedError());
  }
  if (session !== null && state === BackendLifecycleState.Ready) {
    return Promise.resolve(session);
  }
  if (starting === null) {
    starting = start().finally(() => {
      starting = null;
    });
  }
  return starting;
};
```

这个设计比在每个调用方加锁更干净。UI 可以在 runtime 启动时 `ensureStarted()`，用户也可以马上提交 prompt；client 自己保证只有一个 launch in flight。

## request timeout 是 fatal

每个 `submitMessage` 会包一层 request timeout：

```ts
async submitMessage(params: MessageSubmitParams): Promise<MessageSubmitResult> {
  if (disposed) {
    throw disposedError();
  }
  const active = await ensureSession();
  try {
    return await withRequestTimeout(active.client.submitMessage(params), requestTimeoutMs);
  } catch (error) {
    if (isFatalBackendError(error)) {
      markDead();
    }
    throw error;
  }
},
```

为什么 request timeout 要让 client dead？因为对于 stdio JSON-RPC，一个请求超时后，connection 的未来状态已经不可靠：迟到 response 可能还会回来，也可能后端内部已经卡住。U7 不尝试在同一 connection 上继续排队，而是 dispose 当前 backend，并在下一次 submit 时重新 spawn。

## disposal race 必须回收新进程

`start` 里有两段 race guard：launch 结束后、ready 结束后都检查 state 是否仍是 `Starting`。

```ts
if (state !== BackendLifecycleState.Starting) {
  backend.dispose();
  throw abortedError();
}
```

```ts
if (state !== BackendLifecycleState.Starting) {
  connection.dispose();
  backend.dispose();
  throw abortedError();
}
```

为什么要这么严？因为用户可能在后端启动过程中退出 TUI。如果 launch promise 后来才 resolve，client 已经 disposed，此时不能“复活”一个没人持有的 child process。U7 的测试专门覆盖了这个 race：启动中 dispose，随后 launch resolve，必须 dispose 刚得到的 backend。

## dead 后下一次 submit 才 respawn

U7 没有做自动重放，也没有后台静默重启。原因是 in-flight request 可能代表用户输入，自动重放会引入“用户没确认但请求又执行了一次”的风险。更安全的策略是：当前请求失败并展示错误，client 标记 dead；只有下一次用户提交新的非空 prompt 时，才通过 `ensureSession()` 重新启动后端。
