U7 的测试分三层：Rust backend wire-format 测试、TypeScript JSON-RPC client 测试、UI runtime store 测试。这样可以分别证明协议帧正确、client lifecycle 正确、UI seam 不会静默丢 prompt。

![U7 后端客户端测试分层](../../images/u7-backend-client/backend-client-test-layers.png)

## Rust wire-format 测试

[`message_submit.rs`](https://github.com/kefeiqian/KQode/blob/7c9d43fa3af1413967813e51709ee6790151e8aa/tests/message_submit.rs) 新增 ready notification 测试：

```rust
#[test]
fn backend_announces_ready_before_handling_requests() {
    // Close stdin with no request: the backend's only output must be the
    // one-shot ready notification it emits before entering its request loop.
    let output = backend_output(&[]);

    assert!(output.status.success(), "{output:?}");
    let frames = parse_stdout_frames(&output.stdout);
    assert_eq!(
        frames.len(),
        1,
        "startup with no request must emit only the ready notification: {frames:?}"
    );
```

这个测试的价值是把“ready 必须先于任何 request handling”写死。如果以后有人改 Rust backend，把 ready 移到 loop 之后或删掉，测试会立即失败。

## TypeScript fake backend 测试

[`backendClient.test.ts`](https://github.com/kefeiqian/KQode/blob/7c9d43fa3af1413967813e51709ee6790151e8aa/tui/src/backend/client/__tests__/backendClient.test.ts) 用 in-memory `PassThrough` stream 构造 fake backend：

```ts
const server = createMessageConnection(
  new StreamMessageReader(backendStdin),
  new StreamMessageWriter(backendStdout)
);
configure(server);
server.listen();
if (signalReady) {
  void server.sendNotification(backendReadyNotification);
}
```

为什么不用每个 case 都启动真实 Rust 进程？因为 lifecycle 分支很多：ready timeout、recoverable protocol error、hung request、backend exit、dispose race。用 fake backend 可以精确制造这些状态，而且速度更快、失败原因更集中。真实进程只留给少数 integration case。

## hung request 会 kill 当前 backend 并 respawn

测试覆盖了 request timeout 后下一次 submit 重新启动：

```ts
it('times out a hung request, marks the client dead, and respawns on the next submit', async () => {
  const hung = makeFakeBackend((server) =>
    server.onRequest(messageSubmitRequest, () => new Promise<MessageSubmitResult>(() => undefined))
  );
  const healthy = makeFakeBackend(ack);
  const backends = [hung, healthy];
  const launch = vi.fn(async () => backends.shift()?.launched as LaunchedBackend);
  const client = createBackendClient({ launch, requestTimeoutMs: 100 });
```

这个测试同时验证了三件事：timeout 是 fatal、旧 backend 会 dispose、下一次 submit 会通过 `launch` 得到新的 backend。它没有验证自动重放，因为 U7 明确不做自动重放。

## runtime store 测试防止 silent drop

[`backendRuntime.test.ts`](https://github.com/kefeiqian/KQode/blob/7c9d43fa3af1413967813e51709ee6790151e8aa/tui/src/backend/runtime/__tests__/backendRuntime.test.ts) 覆盖 eager start 失败后的提交：

```ts
it('settles a visible error entry for a submit after a failed start (no silent drop)', async () => {
  const store = createStore();
  const failure = new BackendClientError(BackendErrorKind.Launch, 'backend unavailable');
  const client = fakeClient({
    ensureStarted: vi.fn().mockRejectedValue(failure),
    submitMessage: vi.fn().mockRejectedValue(failure)
  });

  startBackendRuntime(store, client);
```

这个测试不是协议测试，而是产品行为测试：后端启动失败后，用户再输入内容时，界面应该出现 error entry，而不是把 prompt 吞掉。对 coding agent 来说，“可见失败”比“看似没反应”更重要。

## 真实 source backend integration

最后，U7 仍保留一条真实 source-mode integration：

```ts
it(
  'starts the Rust backend, submits, and receives the ACK with exact receivedText',
  async () => {
    const client = createSourceBackendClient({ repoRoot, workspaceCwd: repoRoot });
    try {
      const result = await client.submitMessage({ text: '  café\n☕  ' });
      expect(result.message).toBe(ACK_MESSAGE);
      expect(result.receivedText).toBe('  café\n☕  ');
    } finally {
      client.dispose();
    }
  },
  INTEGRATION_TIMEOUT_MS
);
```

它证明 U5 协议、U6 launcher 和 U7 client 串起来后，Unicode、换行和空白都能原样返回。
