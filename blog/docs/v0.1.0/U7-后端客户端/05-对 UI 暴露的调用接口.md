title: 5. 对 UI 暴露的调用接口
---

U7 不让 React / Ink 组件直接 import process launcher 或 `vscode-jsonrpc`。相反，它通过 Jotai store 暴露一个窄 `BackendClient` seam：UI 只知道“提交消息”和“后端启动中提示”，不知道 child process、stdio stream、ready notification 或 request id。

## runtime client 的最小形状

[`backendRuntime.ts`](https://github.com/kefeiqian/KQode/blob/7c9d43fa3af1413967813e51709ee6790151e8aa/tui/src/backend/runtime/backendRuntime.ts) 定义 runtime 需要的 client：

```ts
export type RuntimeBackendClient = BackendClient & {
  ensureStarted(): Promise<void>;
  dispose(): void;
};
```

这里没有暴露 `getState()`，也没有暴露 `MessageConnection`。原因是 UI 层当前只需要发消息、启动和释放。如果组件开始依赖 lifecycle state，很容易把展示逻辑和连接管理耦合起来。更窄的接口让后续替换 source-mode client、packaged client 或测试 fake client 都更容易。

## composition root 注入 client

`startBackendRuntime` 把 client 写入 `backendClientAtom`，并马上启动：

```ts
export function startBackendRuntime(store: Store, client: RuntimeBackendClient): () => void {
  store.set(backendClientAtom, client);
  store.set(startupStatusHintAtom, BACKEND_LOADING_HINT);

  void client
    .ensureStarted()
    .catch(() => {
      // Keep the Dead-state client in the seam (do not dispose or clear it): the
      // next submit retries through ensureSession() and, on repeat failure,
      // settles a visible backend-error entry rather than silently dropping it.
    })
    .finally(() => {
      store.set(startupStatusHintAtom, undefined);
    });
```

为什么 eager start？如果等用户第一次 prompt 才启动，第一次交互会混合“用户请求”和“后端冷启动”。U7 选择在 runtime 开始时提前启动，并通过 `Loading backend` hint 告知用户当前状态。

## 启动失败时保留 dead-state client

`catch` 里没有 dispose，也没有清空 atom。这是 U7 一个很重要的 UX 决策：

```ts
.catch(() => {
  // Keep the Dead-state client in the seam (do not dispose or clear it): the
  // next submit retries through ensureSession() and, on repeat failure,
  // settles a visible backend-error entry rather than silently dropping it.
})
```

如果 eager start 失败后清空 `backendClientAtom`，用户下一次输入可能找不到 backend seam，最终表现成 prompt 被静默丢掉。保留 dead-state client 的好处是：下一次 submit 仍然会走同一个 client，触发 retry；如果仍失败，会生成可见 error entry。

## disposer 清理 seam

函数返回 disposer：

```ts
return () => {
  client.dispose();
  store.set(backendClientAtom, undefined);
};
```

这个方向和 U6 的 `dispose()` 对齐：composition root 拥有 backend lifecycle，退出时统一释放。React 组件不需要知道 process tree 怎么清理，store 也不会长期持有已经退出的 client。

## 为什么这对 UI 架构重要

KQode 的架构目标是 Rust core headless 可运行，TypeScript Ink 只是 rich surface。U7 的 seam 让 UI 层保持“调用业务接口”，而不是“管理 transport”。后续如果加入 packaged backend、daemon mode、session restore 或 MCP 工具，UI 组件仍然可以继续依赖 `BackendClient`，而不是被迫理解每一种后端连接方式。
