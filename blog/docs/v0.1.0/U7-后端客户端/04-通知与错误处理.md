U7 把错误拆成几类：startup readiness timeout、transport close / error、JSON-RPC method error、request timeout、launch failure。这样 UI 后续可以显示更准确的错误，而不是把所有失败都揉成“后端坏了”。

![U7 JSON-RPC client 错误分类](../../images/u7-backend-client/backend-client-error-taxonomy.png)

## readiness timeout

[`waitForBackendReady`](https://github.com/kefeiqian/KQode/blob/7c9d43fa3af1413967813e51709ee6790151e8aa/tui/src/backend/client/backendReadiness.ts) 注册 notification、close、error 三类监听，并设置启动超时：

```ts
const timer = setTimeout(() => {
  settle(() =>
    reject(
      new BackendClientError(
        BackendErrorKind.Timeout,
        `backend did not report JSON-RPC readiness within ${startupTimeoutMs}ms`
      )
    )
  );
}, startupTimeoutMs);
```

为什么 startup timeout 和 request timeout 分开？因为它们定位不同问题。startup timeout 表示“进程启动了但没有进入可通信状态”；request timeout 表示“已经 ready 的 connection 没有及时返回某个请求”。分开后，测试和 UI 都能更精确地判断失败阶段。

## listener 清理避免 late event 污染状态

同一个函数里有 `settle` helper：

```ts
const settle = (action: () => void): void => {
  if (settled) {
    return;
  }
  settled = true;
  clearTimeout(timer);
  for (const registration of registrations) {
    registration.dispose();
  }
  action();
};
```

这段看似模板化，但对 lifecycle 很关键。ready、close、error、timeout 这些事件有竞争关系；第一个事件决定启动结果，后来的 late notification 或 late close 不应该再次修改 promise 状态，也不应该留下监听器泄漏。

## 打开 ready connection 时先接监听，再 listen

[`openReadyConnection`](https://github.com/kefeiqian/KQode/blob/7c9d43fa3af1413967813e51709ee6790151e8aa/tui/src/backend/client/backendReadiness.ts) 的顺序是：创建 connection、注册 fatal listeners、创建 ready promise、`listen()`、等待 ready。

```ts
const connection = createMessageConnection(
  new StreamMessageReader(backend.stdout),
  new StreamMessageWriter(backend.stdin)
);
connection.onClose(onFatal);
connection.onError(onFatal);
backend.onExit(onFatal);

const ready = waitForBackendReady(connection, startupTimeoutMs);
connection.listen();
try {
  await ready;
} catch (error) {
  connection.dispose();
  backend.dispose();
  throw error;
}
```

顺序不能反过来。ready notification 是 one-shot；如果先 `listen()` 再注册 listener，理论上可能错过第一帧。U7 把监听器注册放在前面，保证 ready 信号不会丢。

## JSON-RPC method error 是 protocol error

[`messageConnectionClient.ts`](https://github.com/kefeiqian/KQode/blob/7c9d43fa3af1413967813e51709ee6790151e8aa/tui/src/backend/client/messageConnectionClient.ts) 把 `ResponseError` 转成 `BackendErrorKind.Protocol`：

```ts
if (error instanceof ResponseError) {
  return new BackendClientError(
    BackendErrorKind.Protocol,
    `backend rejected message submit: ${error.message}`,
    { cause: error }
  );
}
```

为什么 protocol error 不一定 fatal？比如 invalid params 是 method 层拒绝，不代表 transport 已经坏掉。U7 的 `isFatalBackendError` 也明确表达了这个取舍：

```ts
export function isFatalBackendError(error: unknown): boolean {
  return error instanceof BackendClientError && error.kind !== BackendErrorKind.Protocol;
}
```

结果是：单个 request 被 backend 拒绝时，进程可以继续服务后续请求；transport error、timeout、launch error 才会让 client 进入 dead。
