U10 还做了一类看似琐碎、实际很关键的重构：把变大的 component/state/helper 文件拆开，把重复或通用的逻辑放进 `libs/`。这和 KQode 项目的 200 行约定直接相关：文件不是越少越好，真正重要的是每个文件的变化理由足够单一。

![文件拆分与公共工具](../../images/u10-architecture-hardening/file-splitting-helpers.png)

## 把 200 行约定写进 TUI 指令

U10 修改了 [`tui/AGENTS.md`](https://github.com/kefeiqian/KQode/blob/20e31d9aac2736f80989529b98d582d7417b6e51/tui/AGENTS.md)，明确要求 TUI source 先拆分再长大：

```md
Split TUI source files into focused modules before they grow beyond 200 lines. Prefer colocated folders under `src/components/` for component modules when a component needs rendering, state, input, or helper pieces.

Prefer Jotai atoms/selectors over drilling HomeScreen config or shared TUI state through multiple component layers. When a leaf component is specific to the stateful TUI shell, read shared screen state from atoms near that leaf instead of threading config props through intermediate wrappers.
```

这两段其实是一条规则的两个侧面：文件要小，状态通道也要窄。否则 component 文件会同时承担 rendering、input、layout、state 转发、backend callback，最后任何一个小改动都会碰到整屏代码。

## 删除根级 `PromptComposer.tsx` wrapper

U9 还有一个根级 [`tui/src/components/PromptComposer.tsx`](https://github.com/kefeiqian/KQode/blob/f642fb3b65bad29da3497d77967af741424205ca/tui/src/components/PromptComposer.tsx) wrapper：

```tsx
export {
  PromptComposer,
  formatVisiblePrompt,
  resolveComposerCursorPosition
} from '@components/PromptComposer/index.js';
```

U10 删除了这个文件，让调用方直接 import folder entry [`tui/src/components/PromptComposer/index.tsx`](https://github.com/kefeiqian/KQode/blob/20e31d9aac2736f80989529b98d582d7417b6e51/tui/src/components/PromptComposer/index.tsx)。这不是为了少一个文件，而是为了让 component folder 成为唯一入口。否则项目里会同时出现：

```ts
@components/PromptComposer.tsx
@components/PromptComposer/index.tsx
@components/PromptComposer/ComposerFrame.tsx
```

这种双入口会让 import path 漂移，后续重构时很难判断哪个文件是 public surface。U10 之后，`PromptComposer/` 文件夹就是组件模块边界。

## composer 文本逻辑从 atom 文件里拆出

U9 的 `composerAtoms.ts` 同时包含 atom、提交校验、UTF-8 byte length、cursor code point 移动等逻辑。U10 把纯文本校验放进 [`tui/src/state/composer/text.ts`](https://github.com/kefeiqian/KQode/blob/20e31d9aac2736f80989529b98d582d7417b6e51/tui/src/state/composer/text.ts)：

```ts
export function validateComposerSubmit(
  text: string,
  maxBytes = PROMPT_MAX_BYTES
): SubmitValidation {
  if (text.trim().length === 0) {
    return {
      ok: false,
      reason: 'empty',
      message: ''
    };
  }

  const limitMessage = overLimitMessage(text, maxBytes);
  if (limitMessage !== null) {
    return {
      ok: false,
      reason: 'over-limit',
      message: limitMessage
    };
  }

  return {
    ok: true,
    text
  };
}
```

这样做的 WHY 是：校验规则和 atom mutation 是两类变化。以后 prompt limit、空白规则、错误文案可能变；cursor state mutation 也可能变。如果它们在同一个文件里，测试和 review 都会把不相关变化混在一起。

## UTF-8 helper 进入 `libs/text/`

U10 新增 [`tui/src/libs/text/utf8.ts`](https://github.com/kefeiqian/KQode/blob/20e31d9aac2736f80989529b98d582d7417b6e51/tui/src/libs/text/utf8.ts)，把字节长度和截断变成通用工具：

```ts
/** UTF-8 byte length of `text`. */
export function utf8ByteLength(text: string): number {
  return textEncoder.encode(text).length;
}
```

为什么不把它留在 composer 里？因为 UTF-8 byte budget 不只会出现在 prompt composer。后续 tool output、context fragment、backend message、trace log 都可能需要按 byte 或 token 预算截断。提前放到 `libs/text/`，让这个行为可测试、可复用、可搜索。

## `clamp` 也从局部 helper 变成公共 math 工具

U9 的 `BodyPane` 自己有局部 `clamp`。U10 新增 [`tui/src/libs/math/clamp.ts`](https://github.com/kefeiqian/KQode/blob/20e31d9aac2736f80989529b98d582d7417b6e51/tui/src/libs/math/clamp.ts)：

```ts
/** Restricts `value` to the inclusive range between `min` and `max`. */
export function clamp(value: number, min: number, max: number): number {
  return Math.min(max, Math.max(min, value));
}
```

然后 [`BodyPane`](https://github.com/kefeiqian/KQode/blob/20e31d9aac2736f80989529b98d582d7417b6e51/tui/src/components/BodyPane.tsx) 与 home screen scroll atom 都可以共享同一个边界规则：

```tsx
const maxScrollOffset = Math.max(0, allRows.length - visibleRows);
const scrollOffset = clamp(resolvedScrollOffsetRows, 0, maxScrollOffset);
```

这个 helper 很小，但它避免了多个文件各写一遍 `Math.min(Math.max())`，也避免了参数顺序或 inclusive/exclusive 语义不一致。

## 拆分不是为了抽象，而是为了让变化局部化

U10 的拆分有一个共同特征：每个新文件都有明确的变化理由。

- `state/global/dimensions.ts`：terminal canvas 与 test override。
- `state/global/statusHint.ts`：startup/loading hint。
- `state/homeScreen/layout.ts`：首页布局数学。
- `state/composer/text.ts`：prompt 文本校验。
- `libs/text/utf8.ts`：UTF-8 byte 工具。
- `libs/math/clamp.ts`：数值边界工具。

这和过早抽象不同。过早抽象通常是把“未来可能复用”的东西做成复杂框架；U10 做的是把已经出现的职责边界切开，让每个文件在接下来继续增长时不会互相拖累。
