## U9 的问题：配置对象开始变成隐形总线

U9 的 [`HomeScreen`](https://github.com/kefeiqian/KQode/blob/f642fb3b65bad29da3497d77967af741424205ca/tui/src/components/HomeScreen/index.tsx) 仍然把整屏配置作为一个 `config` 注入 Jotai store：

```tsx
export type HomeScreenProps = {
  config: HomeScreenConfig;
};

export function HomeScreen({ config }: HomeScreenProps) {
  const store = useHomeScreenStore(config);

  return (
    <Provider store={store}>
      <HomeScreenView />
    </Provider>
  );
}
```

这个写法在 U9 是合理的：它把 props drilling 从叶子组件里拿掉了，也让 `HomeScreenView` 可以读 atom。但是进入 U10 时，它开始暴露三个问题：

1. **状态归属不清楚**。`productVersion`、`workspaceCwd`、`columns`、`rows`、`modelLabel`、`onPromptSubmit` 都塞在一个 `HomeScreenConfig` 里，看起来属于同一个 screen，但实际上有些是全局 runtime 状态，有些是 layout 输入，有些是 backend 提交通道。
2. **测试入口太粗**。测试如果只想固定 `columns`，也要构造整块 config。以后增加 `statusHint`、`inputLocked` 或 backend fake 时，这个 config 会继续膨胀。
3. **后端边界容易被 UI 配置吞掉**。`onPromptSubmit` 放在 screen config 里，会把“用户输入提交到 backend”伪装成一个普通 view callback。对 coding agent 来说，这不是 view callback，而是 runtime 边界。

## U10 的目标：把状态按寿命和责任拆开

U10 的新增导出入口 [`tui/src/state/global/index.ts`](https://github.com/kefeiqian/KQode/blob/20e31d9aac2736f80989529b98d582d7417b6e51/tui/src/state/global/index.ts) 很能说明重构方向：

```ts
export * from '@state/global/backend.ts';
export * from '@state/global/body.ts';
export * from '@state/global/dimensions.ts';
export * from '@state/global/gitStatus.ts';
export * from '@state/global/inputLock.ts';
export * from '@state/global/model.ts';
export * from '@state/global/session.ts';
export * from '@state/global/statusHint.ts';
export * from '@state/global/workspace.ts';
```

这里不是为了“目录更漂亮”。它把原来混在 `HomeScreenConfig` 里的字段，重新按语义放回它们应该在的位置：

- `workspaceCwdAtom` 与 `repoRootAtom` 是 runtime identity。
- `columnsAtom` 与 `rowsAtom` 是 terminal canvas 尺寸。
- `backendClientAtom` 是 backend seam，不是 view callback。
- `statusHintAtom` 与 `inputLockedAtom` 是跨组件 UI 状态。
- `bodyEntriesAtom` 是 transcript/body 输入。

这样一来，`HomeScreen` 不再是“配置对象容器”，而是读取全局状态与 screen-local state 的 composition surface。下一步要接 prompt queue、backend response、session id 或审批状态时，不需要继续往 `HomeScreenConfig` 里塞字段。

## 架构故事：从单屏配置到可演进 shell

U10 的 before → after 可以概括成下面这条线：

```text
U9: App/HomeScreen props
  -> HomeScreenConfig
  -> homeScreenConfigAtom
  -> leaf components read derived screen config

U10: composition root / tests seed narrow atoms
  -> state/global/* owns app-wide facts
  -> state/homeScreen/* owns screen layout and scroll
  -> components read only what they need
```

这个变化的价值要到后续 milestone 才会完全显现。Coding agent TUI 不只是显示一段文本，它会持续处理：窗口 resize、backend startup、prompt 提交、流式响应、Git 状态、输入锁、审批提示、错误状态、session 记录。把这些都挂在一个 config 对象上，短期省事，长期会让每个改动都穿过同一根“巨型管道”。U10 的重构是在管道变粗之前先拆管线。

## 读代码时要注意的边界

U10 里有一个容易误读的点：提交标题提到 backend module architecture，但这次 diff 主要不是重写 `tui/src/backend/` 下的 source / packaged 代码，而是新增一篇架构 solution，并把 shared state 里的 backend seam 固化出来。也就是说，U10 的 backend 加固重点是“边界所有权”和“状态层不要拥有进程生命周期”，而不是重新实现 backend launcher。

这一点可以从新增的 [`backend-process-lifecycle-ownership-in-the-ink-tui.md`](https://github.com/kefeiqian/KQode/blob/20e31d9aac2736f80989529b98d582d7417b6e51/docs/solutions/architecture-patterns/backend-process-lifecycle-ownership-in-the-ink-tui.md) 看出来：它把先前形成的 backend lifecycle 规则沉淀为项目知识，避免后续为了修一个 test 或接一个 atom，又把 process spawn 逻辑塞回 state/component。
