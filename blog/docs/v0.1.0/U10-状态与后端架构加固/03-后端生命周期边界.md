title: 3. 后端生命周期边界
---

U10 的 backend 加固，核心不是在这个 commit 里大规模改写 `backend/` 目录，而是把 backend 进程生命周期的所有权写成项目级架构规则，并在 shared state 中保留一个足够窄的 seam：state 可以持有已经创建好的 client，但不能知道 process 是怎么 spawn、build、materialize 或 dispose 的。

![后端生命周期边界](../../images/u10-architecture-hardening/backend-lifecycle-boundary.png)

## 新增 solution：把隐性架构规则变成可复用知识

U10 新增了 [`docs/solutions/architecture-patterns/backend-process-lifecycle-ownership-in-the-ink-tui.md`](https://github.com/kefeiqian/KQode/blob/20e31d9aac2736f80989529b98d582d7417b6e51/docs/solutions/architecture-patterns/backend-process-lifecycle-ownership-in-the-ink-tui.md)。这篇文档的关键判断是：backend 进程的生命周期等于 TUI 进程生命周期，而不是 React component 生命周期。

原文里的 guidance 很直接：

```md
Own process-backed lifecycles at the **composition root** (`main.tsx`), not in React state or components. State holds only a narrow, already-created client seam.
```

为什么这句话值得单独沉淀？因为在 Ink TUI 里，很容易把 `App` 当成“根”，然后用 `useEffect` 做初始化和 cleanup。但对一个长期运行的 backend child process 来说，React mount/unmount 不是正确的所有权边界。StrictMode、测试 remount、未来多 shell 或热更新，都可能让 component lifecycle 和 process lifecycle 脱钩。

## state 只保存窄 client seam

U10 新增的 [`tui/src/state/global/backend.ts`](https://github.com/kefeiqian/KQode/blob/20e31d9aac2736f80989529b98d582d7417b6e51/tui/src/state/global/backend.ts) 非常克制：

```ts
import { atom } from 'jotai';
import type { BackendClient } from '@contracts/backend/index.ts';

/** Injected backend seam (submitMessage-only) the prompt queue submits through. */
export const backendClientAtom = atom<BackendClient | undefined>(undefined);
```

这里只有 `BackendClient | undefined`，没有 `launchSourceBackend()`，没有 `createProcessBackendClient()`，没有 `node:child_process`。这就是 U10 想守住的边界：state 可以让 prompt queue 找到“提交消息”的接口，但 state 不拥有“如何启动 backend 进程”的知识。

这条边界有两个收益：

1. **测试不 spawn 真实进程**。测试可以往 `backendClientAtom` 注入 fake client，只验证 prompt queue 行为。
2. **部署形态可以变化**。开发态可能从 source build，打包态可能 materialize embedded binary；state 都不需要知道。

## source 与 packaged 是 composition 选择，不是 state 选择

在 U10 的 tree 里，source path 和 packaged path 已经是两条清晰的 client composition 入口。开发态 [`tui/src/backend/client/sourceBackendClient.ts`](https://github.com/kefeiqian/KQode/blob/20e31d9aac2736f80989529b98d582d7417b6e51/tui/src/backend/client/sourceBackendClient.ts) 负责从源码构建并运行：

```ts
export function createSourceBackendClient(options: SourceBackendClientOptions): BackendClientHandle {
  const { repoRoot, workspaceCwd, requestTimeoutMs } = options;
  return createBackendClient({
    launch: () => launchSourceBackend({ repoRoot, workspaceCwd }),
    requestTimeoutMs
  });
}
```

打包态 [`tui/src/backend/client/packagedBackendClient.ts`](https://github.com/kefeiqian/KQode/blob/20e31d9aac2736f80989529b98d582d7417b6e51/tui/src/backend/client/packagedBackendClient.ts) 则走 materialize + spawn：

```ts
export function createPackagedBackendClient(
  options: PackagedBackendClientOptions
): BackendClientHandle {
  return createBackendClient({
    launch: () =>
      launchPackagedBackend({
        asset: options.asset,
        version: options.version,
        workspaceCwd: options.workspaceCwd,
        cacheBaseDir: options.cacheBaseDir
      }),
    requestTimeoutMs: options.requestTimeoutMs
  });
}
```

这两段代码都来自 U10 固定的 commit tree。U10 的重点是：shared state 不应该直接 import 这些模块。source / packaged 是 composition root 的选择，而不是 Jotai atom 的选择。

## packaged backend 为什么要有 materialize 层

[`launchPackagedBackend`](https://github.com/kefeiqian/KQode/blob/20e31d9aac2736f80989529b98d582d7417b6e51/tui/src/backend/packaged/launchPackagedBackend.ts) 的职责也很窄：先 materialize，再用共享 spawn guard 启动。

```ts
export async function launchPackagedBackend(
  options: LaunchPackagedBackendOptions,
  deps: LaunchPackagedBackendDeps = {}
): Promise<LaunchedBackend> {
  const materialize = deps.materialize ?? materializePackagedBackend;
  const spawn = deps.spawn ?? spawnBackend;

  const binaryPath = await materialize({
    asset: options.asset,
    version: options.version,
    cacheBaseDir: options.cacheBaseDir
  });

  return spawn({
    binaryPath,
    workspaceCwd: options.workspaceCwd
  });
}
```

这里的设计点是：打包态不应该调用 Cargo，也不应该让 generic `createBackendClient()` 知道 embedded asset 怎么落盘。`createBackendClient()` 只关心“给我一个 launch function，我管理 JSON-RPC client lifecycle”；packaged 模块关心“如何把 embedded backend 安全变成可执行文件”。

更底层的 [`materializePackagedBackend`](https://github.com/kefeiqian/KQode/blob/20e31d9aac2736f80989529b98d582d7417b6e51/tui/src/backend/packaged/materializeBackend.ts) 把 integrity 和 cache 规则写清楚：

```ts
if (inspectExisting(binaryPath, asset.sha256, platform) === 'reusable') {
  return binaryPath;
}

const bytes = await asset.readBytes();
const actualSha = sha256Hex(bytes);
if (actualSha !== asset.sha256) {
  throw materializationError(
    `embedded asset integrity check failed (expected ${asset.sha256}, got ${actualSha})`
  );
}
```

这也解释了为什么 backend 架构不能混进 state：materialize、cache、integrity、spawn failure、Windows file lock fallback，都是 process/runtime 层的问题。把它们藏在 atom 里，会让 UI state 同时承担文件系统、安全校验和进程管理职责。

## 为什么 U10 要把这个写进 docs/solutions

因为这类架构边界最容易在后续需求压力下被破坏。比如当 prompt 提交失败时，开发者可能想“在 atom 里 lazy start backend”；当测试需要 fake backend 时，可能想“给 component 加一个 `backendClient` prop”；当打包态失败时，可能想“让 state 直接重试 materialize”。这些局部修复都会让边界慢慢腐蚀。

U10 把规则写成 solution 文档，是为了让后续 agent 或人类开发者在改 backend runtime 前先看到一条明确原则：**composition root 拥有进程生命周期，state 只持有窄 seam，components 不 import launch code**。这不是为了讨好测试，而是为了让 source/dev、packaged/dist、unit test 三种运行形态共享同一套 UI state。
