U10 最大的代码层变化，是把共享 Jotai state 从单个 `homeScreenAtoms.ts` 拆成 `state/global/`、`state/homeScreen/` 和 `state/composer/`。这一步的重点不是“多几个文件”，而是把状态按照**寿命**和**责任**重新分层。

![共享状态目录重组](../../images/u10-architecture-hardening/state-global-reorganization.png)

## U9：一个 `HomeScreenConfig` 承担太多角色

U9 的 [`tui/src/state/homeScreenAtoms.ts`](https://github.com/kefeiqian/KQode/blob/f642fb3b65bad29da3497d77967af741424205ca/tui/src/state/homeScreenAtoms.ts) 把 screen 输入集中在一个类型里：

```ts
export type HomeScreenConfig = {
  productVersion: string;
  workspaceCwd: string;
  gitStatusLabel?: string;
  modelLabel: string;
  bodyEntries?: readonly BodyEntry[];
  columns: number;
  rows: number;
  onPromptSubmit: (prompt: string) => void;
};

export const homeScreenConfigAtom = atom<HomeScreenConfig>({
  productVersion: '',
  workspaceCwd: '',
  modelLabel: DEFAULT_MODEL_LABEL,
  columns: DEFAULT_COLUMNS,
  rows: DEFAULT_ROWS,
  onPromptSubmit: noopPromptSubmit
});
```

这个结构的问题是：它把不同层级的事实压扁了。

- `productVersion` 属于 app identity。
- `workspaceCwd` 属于 runtime workspace。
- `columns` 和 `rows` 属于 terminal viewport。
- `gitStatusLabel` 是 derived IO 结果。
- `modelLabel` 是全局状态栏展示。
- `onPromptSubmit` 是 backend 提交边界。
- `bodyEntries` 是 body transcript 输入。

它们被放进同一个 config 后，任何叶子组件想读其中一个字段，都间接依赖整块配置。测试想替换其中一个字段，也要知道整块配置怎么构造。这就是“配置对象变成隐形总线”的典型信号。

## U10：全局事实进入 `state/global/`

U10 新增的 [`tui/src/state/global/workspace.ts`](https://github.com/kefeiqian/KQode/blob/20e31d9aac2736f80989529b98d582d7417b6e51/tui/src/state/global/workspace.ts) 很小，但语义很清楚：

```ts
import { atom } from 'jotai';

export const repoRootAtom = atom<string | undefined>(undefined);
export const productVersionAtom = atom('');
export const workspaceCwdAtom = atom('');
```

`workspaceCwd` 不再是 `HomeScreen` 的 props。它是整个 TUI session 的事实：header、cwd line、Git 状态、backend process 都可能需要它。把它放在 `global` 下，后续任何真正需要这个事实的模块都可以读 atom，而不是让中间组件继续转发。

同理，[`tui/src/state/global/model.ts`](https://github.com/kefeiqian/KQode/blob/20e31d9aac2736f80989529b98d582d7417b6e51/tui/src/state/global/model.ts) 把 model label 拆成自己的 atom：

```ts
import { atom } from 'jotai';

export const DEFAULT_MODEL_LABEL = 'GPT-5.5';
export const modelLabelAtom = atom(DEFAULT_MODEL_LABEL);
```

这看起来只是两行，但它避免了一个后续会很烦的问题：当 model label 未来来自 provider selection、session metadata 或 command palette 时，不需要先改 `HomeScreenConfig`，再改所有 render helper。

## screen-local state 留在 `state/homeScreen/`

不是所有 state 都应该进 `global`。滚动、layout、composer 行高这类只服务首页 shell 的状态，被放进 [`tui/src/state/homeScreen/atoms.ts`](https://github.com/kefeiqian/KQode/blob/20e31d9aac2736f80989529b98d582d7417b6e51/tui/src/state/homeScreen/atoms.ts)：

```ts
export const bodyScrollOffsetRowsAtom = atom(0);
export const composerRowsAtom = atom(DEFAULT_COMPOSER_ROWS);
export const submittedPromptEntriesAtom = atom<BodyEntry[]>([]);

export const displayedBodyEntriesAtom = atom((get) => {
  const submittedPromptEntries = get(submittedPromptEntriesAtom);
  const baseBodyEntries = get(bodyEntriesAtom) ?? DEFAULT_BODY_ENTRIES;

  return submittedPromptEntries.length === 0
    ? baseBodyEntries
    : [...baseBodyEntries, ...submittedPromptEntries];
});
```

这里的拆分标准是：如果 state 描述的是整个 app/session 的事实，就放进 `global`；如果 state 只是首页 screen 如何展示这些事实，就放进 `homeScreen`。例如 `bodyEntriesAtom` 是外部 transcript 输入，属于 `global/body.ts`；但 `bodyScrollOffsetRowsAtom` 是当前 screen 如何滚动显示 transcript，属于 `homeScreen/atoms.ts`。

## layout 计算也从 atoms 文件里拆出

U9 里 `resolveHomeScreenLayout` 和 atom 混在同一个文件。U10 把纯 layout 函数放到 [`tui/src/state/homeScreen/layout.ts`](https://github.com/kefeiqian/KQode/blob/20e31d9aac2736f80989529b98d582d7417b6e51/tui/src/state/homeScreen/layout.ts)：

```ts
export function resolveHomeScreenLayout(
  columns: number,
  rows: number,
  bodyEntryCount = Number.POSITIVE_INFINITY,
  composerRows = DEFAULT_COMPOSER_ROWS,
  cwdRows = 1
): { bodyRows: number; composerVisibleRows: number; cwdRows: number } {
  const headerRows = headerRowCount(columns);
  const resolvedCwdRows = Math.max(1, cwdRows);
  const statusRows = 1;
  const composerErrorReserveRows = COMPOSER_ERROR_RESERVE_ROWS;
  const minBodyRows = 1;
  const fixedRows = headerRows + BODY_CWD_GAP_ROWS + resolvedCwdRows + statusRows;
  const maxComposerVisibleRows = Math.max(
    1,
    rows - fixedRows - COMPOSER_BACKGROUND_PADDING_ROWS - composerErrorReserveRows - minBodyRows
  );
  const maxBodyRows = rows - fixedRows - composerRows;

  return {
    bodyRows: Math.max(1, Math.min(maxBodyRows, bodyEntryCount + 1)),
    composerVisibleRows: maxComposerVisibleRows,
    cwdRows: resolvedCwdRows
  };
}
```

为什么这很重要？因为 layout 是 TUI 最容易回归的地方。`cwd` 行、composer、status bar 都要贴底，body 要能滚动，composer 又会因为换行和 validation error 动态增长。把纯函数拆出来后，测试可以直接验证布局数学，component 只负责渲染。

## 叶子组件直接读需要的 atom

U10 后，[`StatusBar`](https://github.com/kefeiqian/KQode/blob/20e31d9aac2736f80989529b98d582d7417b6e51/tui/src/components/StatusBar.tsx) 不再接收 `columns` 和 `modelLabel` props，而是直接读取共享 atom：

```tsx
export function StatusBar() {
  const columns = useAtomValue(columnsAtom);
  const modelLabel = useAtomValue(modelLabelAtom);
  const statusHint = useAtomValue(statusHintAtom);
  const loadingFrame = useLoadingFrame(statusHint?.kind === 'loading');
```

这不是反模式。对 KQode 这个 stateful TUI shell 来说，`StatusBar` 本来就是读取全局 UI 状态的叶子。让中间的 `HomeScreenView` 代传这些值，反而会制造虚假的 component API，让每次新增全局状态都要改一串 props。

## 为什么不是全部都塞到一个 store initializer

一个可能的替代方案是保留一个 `createHomeScreenConfig()`，只是在内部把它拆成多个 atom。U10 没有这样做，是因为 initializer 仍然会成为中心化入口。后续测试、runtime bootstrap、backend runtime、window resize 都会争着往这个 initializer 里加字段。

U10 的选择是让 composition root 或测试直接 seed 它们真正关心的 atom。这样每条状态边界都显式存在，读代码时也更容易回答：这个值是谁拥有的？谁可以改？它是全局事实，还是 screen-local 展示状态？
