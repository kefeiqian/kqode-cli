title: 6. U10 总结
---

U10 是一次典型的架构加固型 milestone：用户界面看起来变化不大，但代码从“能工作”往“能继续安全演进”迈了一步。它拆 shared state、固化 backend lifecycle 边界、命名测试 seam，并把已经开始膨胀的 helper 逻辑拆进更小的模块。

![U10 总结](../../images/u10-architecture-hardening/summary.png)

## 决策一：用 `state/global/` 承载 session 级事实

U10 把全局状态拆成多个小文件，并通过 [`tui/src/state/global/index.ts`](https://github.com/kefeiqian/KQode/blob/20e31d9aac2736f80989529b98d582d7417b6e51/tui/src/state/global/index.ts) 汇出：

```ts
export * from '@state/global/backend.ts';
export * from '@state/global/body.ts';
export * from '@state/global/dimensions.ts';
export * from '@state/global/gitStatus.ts';
export * from '@state/global/inputLock.ts';
export * from '@state/global/model.ts';
export * from '@state/global/session.ts';
export * from '@state/global/statusHint.ts';
export * from '@state/global/workspace.ts';
```

**为什么：** U9 的 `HomeScreenConfig` 已经开始混合 workspace、viewport、model、body、submit callback。继续沿用会让新增 backend/session/status 状态都经过一个巨型 config。U10 改成按事实归属拆分，让后续 runtime bootstrap 和测试可以只 seed 自己需要的 atom。

## 决策二：backend lifecycle 属于 composition root

U10 新增的 solution 文档明确要求 process-backed lifecycle 由 composition root 拥有，而不是由 React state 或 component 拥有：

```md
Own process-backed lifecycles at the **composition root** (`main.tsx`), not in React state or components. State holds only a narrow, already-created client seam.
```

**为什么：** backend child process 的生命周期和 TUI process 绑定，而不是和某个 component mount 绑定。把 spawn/build/materialize 放进 atom，会破坏测试隔离，也会让 source/dev 与 packaged/dist 两种后端形态泄漏到 UI state。

## 决策三：测试 seam 必须显式命名为 `TestOverride`

[`dimensions.ts`](https://github.com/kefeiqian/KQode/blob/20e31d9aac2736f80989529b98d582d7417b6e51/tui/src/state/global/dimensions.ts) 展示了这个约定：

```ts
export const columnsTestOverrideAtom = /* @__PURE__ */ atom<number | undefined>(undefined);
export const rowsTestOverrideAtom = /* @__PURE__ */ atom<number | undefined>(undefined);

export const columnsAtom = atom((get) => {
  const override = __TEST__ ? get(columnsTestOverrideAtom) : undefined;
  return override ?? get(windowColumnsAtom) ?? DEFAULT_COLUMNS;
});
```

**为什么：** 测试确实需要固定 viewport 和 Git label，但这些入口不能伪装成生产状态。`TestOverride` 后缀 + `__TEST__` gate + pure annotation，让测试 seam 可搜索、可 review，也降低 production path 被测试逻辑污染的风险。

## 决策四：把 layout、text、math helper 拆成可测试模块

U10 把 prompt 文本校验拆进 [`state/composer/text.ts`](https://github.com/kefeiqian/KQode/blob/20e31d9aac2736f80989529b98d582d7417b6e51/tui/src/state/composer/text.ts)，把 UTF-8 helper 拆进 [`libs/text/utf8.ts`](https://github.com/kefeiqian/KQode/blob/20e31d9aac2736f80989529b98d582d7417b6e51/tui/src/libs/text/utf8.ts)：

```ts
export function overLimitMessage(text: string, maxBytes: number): string | null {
  const byteLength = utf8ByteLength(text);
  if (byteLength <= maxBytes) {
    return null;
  }

  return `Prompt is ${byteLength} bytes; maximum is ${maxBytes} bytes.`;
}
```

**为什么：** TUI 的行为细节会越来越多：宽度、换行、UTF-8 byte limit、cursor row、scroll offset、backend error display。把这些规则留在 component 里，会让 UI 文件变成“所有问题的集合”。拆成小 helper 后，每个规则都能单独测试、单独 review。

## 决策五：把 TUI source organization 写成项目规则

U10 更新 [`tui/AGENTS.md`](https://github.com/kefeiqian/KQode/blob/20e31d9aac2736f80989529b98d582d7417b6e51/tui/AGENTS.md)，明确要求文件在超过 200 行前拆分，并优先用 atoms/selectors 代替层层 props drilling。

```md
Split TUI source files into focused modules before they grow beyond 200 lines.

Prefer Jotai atoms/selectors over drilling HomeScreen config or shared TUI state through multiple component layers.
```

**为什么：** 这类规则如果只存在于一次 refactor 的脑子里，下一次 feature rush 时很容易被忘掉。写进 `AGENTS.md` 后，后续 coding agent 在 TUI 目录工作时会优先继承这个约束。

## 延期项

U10 没有试图一次性解决所有问题，几个方向被有意留给后续 milestone：

- **没有重写 backend launcher。** source backend、packaged backend、cache materialize 的结构在 U10 tree 中已经存在；这次主要是固化生命周期边界和 state seam，不把 process ownership 拉回 UI state。
- **没有引入完整 session store。** `state/global/session.ts` 在后续 tree 中会成为 session 事实的入口，但 U10 的主要工作仍是 TUI shell state 的拆分。
- **没有做视觉大改。** status bar、body、composer 的行为更稳，但 U10 不是设计 milestone。视觉 polish 应该在 layout 边界稳定后继续做。
- **没有运行 blog build。** 本篇文档按用户要求只写 Markdown，不执行 `cargo xtask blog-build`。

## 一句话总结

U10 的价值，是在功能继续叠加之前先把“谁拥有状态、谁拥有进程、谁只是测试 seam、哪些 helper 应该独立”讲清楚并写进代码结构。对于一个 coding agent harness 来说，这类重构不显眼，但它决定了后面接 backend、prompt、session、trace 和 eval 时，系统是继续长成清晰的 runtime，还是退化成一个越来越大的 UI 配置对象。
