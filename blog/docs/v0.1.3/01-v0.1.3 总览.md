v0.1.3 是一次围绕“如何真正把 KQode 发给用户”的发布。v0.1.2 已经有可打包的 TUI，可是“打包成功”和“用户稳定安装”之间还有一段距离：发布资产在 GitHub Release，npm 用户期望 `npm install -g @kqode/kqode-cli`，CI 需要在 tag 发布后自动接上 npm，版本号还要在 Rust、TUI、launcher 和平台包之间保持一致。

这一版的主线是把 npm 渠道从“安装时再下载 GitHub Release 二进制”的想法，改成更像 esbuild / SWC 的“launcher + per-platform optional dependency”模型。这样做的结果是：用户安装 npm 包时只接触 npm registry；平台二进制已经在对应平台包里；`kqode` 命令只负责解析并启动本机已安装的可执行文件。

![v0.1.3 发布主线：GitHub Release 到 npm 平台包](../images/v0-1-3-overview/release-flow.png)

## 本次交付

- npm launcher 不再在安装或首次运行时下载二进制，而是通过 `optionalDependencies` 选择平台包。
- `SUPPORTED_TARGETS` 成为运行时解析、平台包生成和 CI 数量校验共同使用的目标列表。
- 新增 `generate-platform-packages.cjs`，在发布时从 GitHub Release archive 生成五个平台包。
- npm 发布工作流先发布平台包，再发布 launcher，并通过 `workflow_run` 在 Release workflow 成功后自动运行。
- `cargo xtask set-version` 同步更新 launcher 自身版本和平台包 pin，避免 tag、Cargo 版本、TUI 版本、npm 依赖版本漂移。
- TUI 入口接入 `citty`，让 packaged `kqode` 可响应 `--help` / `--version`。
- 修正 Docusaurus 的 GitHub Pages `baseUrl` / repo name，并在 `README` 补上 development blog 入口。
- [`f7154952`](https://github.com/kefeiqian/KQode/commit/f7154952) 把最终版本从 `0.1.2` bump 到 `0.1.3`。

## 篇目

1. 本篇：v0.1.3 总览。
2. npm 平台包解析模型：为什么 launcher 只解析本地 package，不再下载。
3. npm 平台包生成器：为什么平台包在发布时生成，而不是提交进仓库。
4. npm CI 发布流：为什么 `npm-publish.yml` 必须作为 top-level workflow 存在。
5. 版本与发布文档：为什么 `set-version` 必须同时 bump `optionalDependencies`。
6. citty CLI：为什么 `--version` / `--help` 落在 TUI entry，而不是 Rust binary。
7. Blog 和 README：为什么部署 URL 也属于 release correctness。
8. v0.1.3 总结：本轮决策、取舍和延期项。
