title: 8. v0.1.3 总结
---

v0.1.3 的主题不是新增 agent 能力，而是把 KQode 的发布链路做得更像一个可以被普通用户安装的 CLI。这个版本的价值在于：它把“构建出来一个二进制”推进到“GitHub Release、npm、版本号、文档入口都能围绕同一个 release 协同”。

![v0.1.3 决策总结](../images/v0-1-3-summary/decision-map.png)

## 决策一：npm 使用 launcher + 平台包，而不是 install-time download

**决定**：`@kqode/kqode-cli` 只作为 launcher，平台 binary 放进 `@kqode/kqode-cli-<platform>-<arch>` 包，并通过 `optionalDependencies` 安装。

**为什么**：install-time download 把用户安装路径暴露给 GitHub Release availability、代理、脚本执行策略和解压差异。平台包模型把复杂度移到发布端，用户端只依赖 npm registry。对 CLI 来说，安装确定性比“少发布几个包”更重要。

**代价**：每个版本要发布六个 npm 包，每个包都需要 npm Trusted Publisher 配置和首次 bootstrap。

## 决策二：SUPPORTED_TARGETS 成为单一事实来源

**决定**：运行时解析、生成器遍历和 CI 数量检查都引用 `SUPPORTED_TARGETS`。

**为什么**：平台矩阵会变化。把目标列表写在多个地方，短期看简单，长期一定漂移。CI 从 `SUPPORTED_TARGETS.length` 派生 expected count，可以让新增或删除平台时失败模式更明确。

**代价**：Node 脚本和 workflow 之间有一条显式依赖：workflow 需要能执行 `node -p "require(...).SUPPORTED_TARGETS.length"`。这是可接受的，因为 npm publish 本来就是 Node / npm 环境。

## 决策三：平台包从 Release archive 生成

**决定**：`generate-platform-packages.cjs` 从 GitHub Release archive 和 `.sha256` 生成临时 npm package 目录。

**为什么**：GitHub Release archive 是已经构建、校验、发布给 direct download 用户的 artifact。npm 平台包应该承载同一份 executable，而不是建立第二套 binary 来源。生成前重新校验 checksum，可以证明 npm 包确实来自 release artifact。

**代价**：npm publish 必须发生在 Release workflow 上传资产之后，不能提前并行。

## 决策四：npm-publish.yml 保持 top-level workflow

**决定**：使用 `workflow_run` 自动衔接 Release workflow，而不是让 `release.yml` 通过 `workflow_call` 调用 npm publish。

**为什么**：npm Trusted Publishing 校验 top-level workflow filename。npm 包配置的是 `npm-publish.yml`；如果 publish 逻辑被 `release.yml` 间接调用，OIDC identity 会变成 `release.yml`，从而失败。这里的设计不是 YAML 风格偏好，而是供应链身份约束。

**代价**：发布逻辑分成两个 workflow，需要用 `workflow_run.head_sha` 和 tag 解析确保 checkout 的是 released commit。

## 决策五：set-version 同步平台包 pin

**决定**：`cargo xtask set-version` 同时改 Rust manifest、TUI manifest、launcher version 和 `@kqode/kqode-cli-*` optional dependency pins。

**为什么**：release tag、packaged binary version、npm launcher version 和平台包 version 必须一致。让维护者手动改五个 dependency pin，迟早会漏；漏掉后用户安装到旧 binary，就是发布事故。

**代价**：`set-version` 里出现了 npm launcher 的特殊规则。v0.1.3 通过 `NPM_MAIN_MANIFEST` 和测试把这条规则显式化。

## 决策六：--help / --version 由 TUI entry 处理

**决定**：用 `citty` 在 TypeScript TUI entry 处理 `--help` / `--version`。

**为什么**：npm `kqode` 命令最终启动的是 packaged Bun / Ink executable，不是 Rust binary。把 meta flag 写在 Rust 入口不会覆盖 npm 分发路径。`citty` 还能避免 source entry 和 packaged entry 各自手写 argv 判断。

**代价**：TUI package 增加一个依赖；但它换来的是可扩展 command structure。

## 决策七：文档 URL 修正也算 release 工作

**决定**：修正 Docusaurus `baseUrl` / repo name，并在 `README` 增加 development blog 入口。

**为什么**：KQode 处在公开构建阶段，文档站是理解项目方向的重要入口。发布后用户访问不到 blog，和安装不到 binary 一样，都会破坏项目可信度。

## 明确延期的内容

- 不在 v0.1.3 自动提交 Homebrew tap 或 winget manifest；文档只说明手工注册方式。
- 不做 code signing、notarization、auto-update。
- 不新增复杂 CLI subcommand，只先补 `--help` / `--version`。
- 不把平台包产物提交进仓库；产物只在发布时生成并发布。
- 不把 npm publish 做成 `release.yml` 内部 step；为了 OIDC 身份稳定，保持独立 top-level workflow。

## 这一版之后的状态

v0.1.3 之后，KQode 的安装故事更完整了：GitHub Release 仍然提供 direct download；npm 提供 self-contained global install；文档说明 Homebrew 和 winget 的后续注册路径；`kqode --version` 能从 packaged TUI 正确返回版本。下一阶段可以继续推进 agent runtime 能力，而不用每次发布时重新手工拼接分发链路。
