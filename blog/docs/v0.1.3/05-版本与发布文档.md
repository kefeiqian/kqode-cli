平台包模型引入后，版本号不再只是 `Cargo.toml` 里的一个字段。v0.1.3 通过 [`540cff96`](https://github.com/kefeiqian/KQode/commit/540cff96) 更新 `cargo xtask set-version`，又通过 [`380cdc1d`](https://github.com/kefeiqian/KQode/commit/380cdc1d) 和后续文档补充，把发布流程写进 [`docs/release/kqode_distribution_registration.md`](https://github.com/kefeiqian/KQode/blob/40a32c53/docs/release/kqode_distribution_registration.md)。

## 为什么 set-version 要理解 npm launcher

在 v0.1.3 以前，`set-version` 只需要改几个 manifest 的 top-level version。平台包模型出现后，launcher 的 `optionalDependencies` 也成了版本契约。最终 release commit [`f7154952`](https://github.com/kefeiqian/KQode/commit/f7154952) 中的 [`packaging/npm/kqode/package.json`](https://github.com/kefeiqian/KQode/blob/f7154952/packaging/npm/kqode/package.json) 是这样的：

```json
{
  "name": "@kqode/kqode-cli",
  "version": "0.1.3",
  "optionalDependencies": {
    "@kqode/kqode-cli-darwin-arm64": "0.1.3",
    "@kqode/kqode-cli-linux-arm64": "0.1.3",
    "@kqode/kqode-cli-linux-x64": "0.1.3",
    "@kqode/kqode-cli-win32-arm64": "0.1.3",
    "@kqode/kqode-cli-win32-x64": "0.1.3"
  }
}
```

如果这里只改 top-level `"version"`，而不改 platform package pins，npm install 会安装旧平台包。也就是说，版本 bump 不是文本替换，而是 release graph 的一致性维护。

## NPM_MAIN_MANIFEST：把特殊规则显式命名

[`xtask/src/support/version.rs`](https://github.com/kefeiqian/KQode/blob/540cff96/xtask/src/support/version.rs) 新增了专门的 launcher manifest 常量：

```rust
/// The launcher package, whose top-level version AND its `@kqode/kqode-cli-*`
/// optional-dependency pins must move together so npm resolves the matching
/// platform packages for the released version.
const NPM_MAIN_MANIFEST: &str = "packaging/npm/kqode/package.json";
```

为什么不把它放进普通 `JSON_MANIFESTS`？因为普通 JSON manifest 只需要改 top-level version；launcher manifest 还要改 dependency pin。把它命名为 `NPM_MAIN_MANIFEST`，可以让未来读代码的人知道这里有 npm 平台包模型带来的特殊规则。

## set_npm_main_version：先改自身，再改依赖

同一个文件里的核心逻辑是：

```rust
fn set_npm_main_version(contents: &str, version: &str) -> Result<String, String> {
    let with_top_level = set_json_version(contents, version)?;
    Ok(set_platform_dependency_pins(&with_top_level, version))
}

fn set_platform_dependency_pins(contents: &str, version: &str) -> String {
    let lines = contents
        .lines()
        .map(|line| {
            if is_platform_dependency_line(line) {
                replace_json_value(line, version)
            } else {
                line.to_string()
            }
        })
        .collect::<Vec<_>>();
    rejoin(contents, &lines)
}
```

这个实现故意保持简单：它没有引入 JSON parser，而是复用原来 line-based rewrite。为什么可接受？因为目标文件是项目自己维护的 `package.json`，结构稳定；函数只替换形如 `"@kqode/kqode-cli-...": "..."` 的依赖行，并用测试保护“不要碰 package 自己的 name 和 URL”。

测试里的意图也很清楚：

```rust
#[test]
fn set_npm_main_version_bumps_top_level_and_platform_pins() {
    let input = "{\n  \"name\": \"@kqode/kqode-cli\",\n  \"version\": \"0.1.2\",\n  \"homepage\": \"https://github.com/kefeiqian/kqode-cli\",\n  \"optionalDependencies\": {\n    \"@kqode/kqode-cli-linux-x64\": \"0.1.2\",\n    \"@kqode/kqode-cli-win32-x64\": \"0.1.2\"\n  }\n}\n";
    let output = set_npm_main_version(input, "0.2.0").unwrap();
    assert!(output.contains("\"version\": \"0.2.0\""));
    assert!(output.contains("\"@kqode/kqode-cli-linux-x64\": \"0.2.0\""));
    assert!(output.contains("\"@kqode/kqode-cli-win32-x64\": \"0.2.0\""));
}
```

## 发布文档把“怎么做”和“为什么”放在一起

[`docs/release/kqode_distribution_registration.md`](https://github.com/kefeiqian/KQode/blob/40a32c53/docs/release/kqode_distribution_registration.md) 不只是操作手册，也记录模型边界：

```md
npm distributes the CLI as a **launcher plus per-platform packages** (the model
esbuild and SWC use), all under the `@kqode` org:

- `@kqode/kqode-cli` — the launcher. It ships no binary and lists one platform
  package per target under `optionalDependencies`.
- `@kqode/kqode-cli-<platform>-<arch>` — five packages (`darwin-arm64`,
  `linux-arm64`, `linux-x64`, `win32-arm64`, `win32-x64`), each carrying that
  host's self-contained executable and declaring `os`/`cpu`.
```

文档里还明确了延期项：Homebrew tap、winget 提交、签名、公证和 auto-update 都不在这一版自动化范围内。这个边界很重要，因为 v0.1.3 的目标不是“所有渠道全自动”，而是先把 GitHub Release 和 npm 这条主链路做成可重复、可解释、可恢复的发布路径。
