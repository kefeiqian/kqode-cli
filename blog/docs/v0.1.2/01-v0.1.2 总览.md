v0.1.2 是一个很小、但对后续发布和协作很关键的版本。它没有扩展 agent loop、工具系统或 TUI 交互，而是先补齐两个更底层的问题：项目应该用什么许可证开放出去，以及 release 操作应该怎样从“人工记步骤”变成“技能按流程推进”。

这两个改动看起来不像功能，但它们决定了 KQode 以后能不能被别人放心试用、二次分发、打包进 npm，乃至被 agent 自动维护。对一个 Rust-first coding-agent harness 来说，早期把这些边界写清楚，比等到生态开始使用后再补要便宜得多。

![v0.1.2 发布总览](../images/v0-1-2-overview/release-overview.png)

## 这个版本交付了什么

v0.1.2 主要交付三件事：

- 项目改为 `MIT OR Apache-2.0` 双许可证，并把 Rust manifest、npm package manifest、README 和许可证正文一起补齐。
- `kqode-version-bump` 技能从“只改版本并提醒后续步骤”升级为“自动 commit + tag，然后分别确认 push 和 npm 发布”。
- release commit 把 root crate、`xtask` crate、TUI package、npm 包和 `Cargo.lock` 里的版本统一提升到 `0.1.2`。

这里的核心思路是：**发布前置条件要尽量进入仓库和自动化流程，而不是留在人脑里。** 许可证如果只写在 README，package manifest 里没有，发布渠道就读不到；版本 bump 如果只改文件、不生成 tag，release pipeline 就不会自然触发；npm 如果在 GitHub Release assets 上传之前发布，用户安装时的 postinstall 又可能拿不到对应二进制。

## 为什么这个小版本值得单独记录

KQode 的目标不是只做一个本地 demo，而是逐步变成可以发布、安装、回放、评测和由 agent 维护的开发者工具。越是底层的工程约定，越应该在早期变成可检查的文件和流程。

许可证是开放协作的契约：别人能不能复制、修改、分发，企业用户能不能接受，贡献者提交代码后默认按什么条款进入项目，都不能靠口头约定。发布流程是分发可信度的契约：tag 对应哪个版本，GitHub Release assets 什么时候生成，npm 包什么时候指向新版本，也不能靠发布当天临时回忆。

所以 v0.1.2 的重点不是“功能多”，而是把项目从“能继续写代码”推进到“更适合作为开源工具持续发布”。
