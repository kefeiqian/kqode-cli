v0.1.2 是一个 release hygiene 版本。它没有推进 agent runtime 的能力边界，却把两个长期会影响协作和分发的问题落到了仓库里：许可证，以及版本发布技能。

最后的 release commit [`26db09b899435b8fe2febfa108395364b13d4bf8`](https://github.com/kefeiqian/KQode/commit/26db09b899435b8fe2febfa108395364b13d4bf8) 把多个 manifest 从 `0.1.1` 提升到 `0.1.2`，包括 root [`Cargo.toml`](https://github.com/kefeiqian/KQode/blob/26db09b899435b8fe2febfa108395364b13d4bf8/Cargo.toml)、[`xtask/Cargo.toml`](https://github.com/kefeiqian/KQode/blob/26db09b899435b8fe2febfa108395364b13d4bf8/xtask/Cargo.toml)、[`tui/package.json`](https://github.com/kefeiqian/KQode/blob/26db09b899435b8fe2febfa108395364b13d4bf8/tui/package.json)、[`packaging/npm/kqode/package.json`](https://github.com/kefeiqian/KQode/blob/26db09b899435b8fe2febfa108395364b13d4bf8/packaging/npm/kqode/package.json) 和 [`Cargo.lock`](https://github.com/kefeiqian/KQode/blob/26db09b899435b8fe2febfa108395364b13d4bf8/Cargo.lock)。

## 决策一：采用 MIT OR Apache-2.0 双许可证

**做了什么：** commit [`4ac3aed7`](https://github.com/kefeiqian/KQode/commit/4ac3aed7eae5623858d578fdd215acf83cdbe29c) 在 Rust manifest、package.json、README、root license 文件和 npm package license 文件里统一声明 `MIT OR Apache-2.0`。

**为什么：** KQode 的核心方向是 Rust-first，而 `MIT OR Apache-2.0` 是 Rust 生态非常常见的 permissive 双许可证组合。MIT 给使用者简单、宽松、兼容性强的路径；Apache-2.0 给需要专利授权和专利终止条款的组织更明确的法律边界。使用 `OR` 而不是 `AND`，是为了让下游按自身项目需要任选其一，降低采用门槛。

**为什么现在做：** 许可证越早确定越好。等项目已经有外部贡献、npm 下载、Release assets 或下游集成之后再补，成本会变高，也容易出现“某些文件有没有被授权清楚”的历史问题。v0.1.2 仍处在早期，因此这是一个低成本、高确定性的改动。

## 决策二：许可证同时写给人和工具

**做了什么：** README 新增 License section；Cargo 和 npm manifest 新增 `license` 字段；npm package 的 `files` allowlist 加入 `LICENSE-APACHE` 和 `LICENSE-MIT`。

**为什么：** README 是给人类读的，manifest 是给工具链、registry、审计工具和包管理器读的，npm tarball 则是用户真正安装到本地的 artifact。只改其中一个位置都不完整。比如 README 有 license，但 npm package 没有 license metadata，registry 页面和合规扫描就可能读不到；仓库有 license 文件，但 package allowlist 没带进去，用户安装包里就缺正文。

**取舍：** 这次没有引入 CLA、DCO、NOTICE 或更复杂的治理文件。原因是当前阶段还没有复杂第三方代码复制或大型贡献者治理需求。先把最必要、最标准的 permissive 许可证链路打通，比提前引入重治理流程更符合项目阶段。

## 决策三：version-bump 自动创建本地 release commit 和 tag

**做了什么：** commit [`3338b530`](https://github.com/kefeiqian/KQode/commit/3338b53016a8dc1662bf97a2459ee3f46961526f) 修改 `kqode-version-bump` skill，让它在 `cargo xtask set-version <chosen-version>` 成功后自动执行：

```bash
git commit -am "chore: release v<chosen-version>"
git tag v<chosen-version>
```

**为什么：** release version 和 git tag 必须一致。这个关系如果靠人手动维护，很容易出错；而本地 commit 和 tag 又是确定性的、可检查的步骤，非常适合交给 skill 自动执行。自动化以后，版本 bump 的产物天然形成一个 release commit，并且 tag 直接指向该 commit。

**安全边界：** skill 仍然要求先确认 working tree 只有 bump 相关变化，并且如果 tag 已存在就停止而不是移动 tag。这两个约束保证自动化不会把无关改动混进 release，也不会破坏已经发布过的坐标。

## 决策四：push 和 npm 发布仍然需要确认

**做了什么：** skill 自动 commit/tag 后，会询问是否 push commit 和 tag；release pipeline 成功后，再询问是否 dispatch `npm-publish.yml`。

**为什么：** push tag 和 npm 发布都是外部副作用。push `v*` tag 会触发 GitHub Actions release pipeline，开始构建并上传公开 release assets；npm 发布会影响所有 `npm install -g @kqode/kqode-cli` 的用户。这些动作应该由维护者明确确认，而不是因为本地 bump 自动发生。

**为什么 npm 要等 release.yml：** npm 包的 postinstall 会下载对应 tag 的 release archive。如果 npm package 先发布，而 GitHub Release assets 还没有上传完成，用户安装新版本时可能失败。因此 skill 明确要求：先 watch `release.yml` 成功，再通过 `npm-publish.yml` 发布 npm。

## 决策五：release commit 只 bump 版本

**做了什么：** `26db09b8` 只把版本字段从 `0.1.1` 提升到 `0.1.2`，并保留前一 commit 添加的 `license = "MIT OR Apache-2.0"` 和 package license metadata。

**为什么：** release commit 应该尽量机械、可审计、低噪音。真正的行为改动在前面的 commit 里完成，最后一个 release commit 只负责让 manifest 和 tag 对齐。这样回看历史时，可以很清楚地区分“功能/流程改变”和“发布版本 bump”。

## 暂缓事项

- 暂缓新增 runtime 功能。v0.1.2 只处理许可证和 release 流程，没有推进 agent loop、provider adapter、VFS、sandbox 或 TUI 功能，因为这些不属于本版本的 release hygiene 目标。
- 暂缓新增复杂开源治理文件。当前只需要 `MIT OR Apache-2.0`、license texts 和 contribution licensing statement；更复杂的 `NOTICE`、CLA 或 DCO 可以等项目规模和依赖形态需要时再补。
- 暂缓把 npm 发布做成无确认全自动。原因是 npm 是公开分发渠道，而且依赖 GitHub Release assets 先完成上传。保留确认步骤可以避免 release pipeline 失败时仍然发布 npm。
- 暂缓改变底层 `cargo xtask set-version` 的职责。version-bump skill 仍然把版本字段修改委托给 `xtask`，而不是自己编辑 manifest。这样单一事实来源仍然在 Rust 自动化命令里，skill 只编排发布步骤。

## 下一步

v0.1.2 之后，KQode 已经有了更清楚的开源授权和更稳的发布交接流程。后续版本可以把注意力重新放回产品能力：本地 terminal agent 的 agent loop、工具注册、文件修改路径、sandbox-lite 执行和可回放 session trace。等这些功能开始对外发布时，v0.1.2 做的 license 和 release 基础设施会继续作为底座。
