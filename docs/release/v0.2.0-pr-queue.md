# v0.2.0 PR Queue Ledger

This ledger tracks the rolling v0.2.0 split. It is operational state; the plan remains the decision artifact.

## Process Rules

- Open exactly one PR at a time.
- Create extraction worktrees outside the repository root, preferably under a sibling directory such as `../KQode.release-split-worktrees/`.
- Create each branch from freshly fetched `origin/main`.
- Record the base SHA before requesting review.
- Run full CI-equivalent checks before opening a PR.
- Local validation on Windows: `cargo test --workspace` can hang when a concurrent dev TUI holds `target\debug\kqode.exe` (backend-spawn integration tests block indefinitely). For fast signal run targeted `cargo test --lib -- --test-threads=1` plus `cargo test -p xtask`, and stream long-running output directly — never pipe it through buffering filters such as `Select-String`, so a hang stays visible. The full CI-equivalent suite still gates the PR on a clean runner.
- Stop for explicit approval before changing queue order or PR boundaries.
- Do not create a generic catch-all polish PR; split leftovers into single-purpose PRs or fold them into a parent PR.
- Extract the final release behavior directly. Do not create a queue PR that introduces an intermediate implementation which a later source commit removes or replaces; hunk-split the source history so the earliest owning PR lands the approved final state.
- All xtask changes ship in PR 01 (`feat/v0.2.0-bootstrap-foundation`); exclude xtask hunks (the `cargo xtask` alias, launcher scripts, `.run/` profiles, `tui-dev`/`tui-dev-here`, and TUI arg forwarding) from every other queue PR.
- Every line in a queue PR must be exercisable end to end through the TUI. Do not ship inert or speculative plumbing (unused protocol fields, unreachable client stacks, dead state atoms); defer such code to the queue PR that wires its real producer/consumer and makes it TUI-testable.
